# 算法基线测试系统 —— 详细设计文档（LLD）

> 文档状态：首版完整设计稿  
> 编写依据：SRS、调研报告、HLD 及最新 `lld_template.md`  
> 适用范围：本地单用户、CLI 手动启动的算法基线测试系统

本文严格保留模板规定的六个一级章节。存储、暂停恢复、异常处理、CLI 和报告等内容放入对应模块的小节中，不再增加新的一级章节。

本设计遵循以下已确认原则：

- 六个章节共同覆盖构建编译、代码版本记录、数据集管理、算法运行、结果评估、回归对比以及结果存储与报告；
- 第一版具体描述 sfvision 和 vloc 两个算法，但公共流程不包含算法专用判断；
- 当前数据集类型处理器实现 RK3399、RK3588 和 KITTI Odometry；EuRoC 保留处理器扩展接口，用户只选择已实现类型，不填写文件匹配、字段解析或分段规则；
- 一次扫描只采用一种数据集类型，同一根目录下的数据按照该类型的内置规则识别；
- 每个有效 Segment 是独立的运行、评估和结果保存单元；检查点在数据集处理完成后推进；
- 数据集参考文件由类型处理器自动发现并写入数据实例配置，voeval 工作流由所选算法的内置契约确定；
- Git 只用于读取和记录代码状态，系统不切换分支、不提交代码；
- 每个 Segment 运行结束后立即评估并保存算法原始输出和评估数据，选定数据写入 run 级 Excel；单个 Segment 或数据集完成时不生成报告，整次 run 结束后才生成汇总运行报告；
- 回归对比只用于同一算法的不同 run，不对不同算法给出“改进”或“退化”结论。

## 一、算法插件配置

每个算法首次接入时填写一份 YAML，以后直接复用。普通用户只选择一次算法，并提供算法路径和编译脚本路径；运行入口、参数映射、固定输出和评估方式由系统内置的算法契约提供。

最小用户配置如下：

```yaml
algorithm: sfvision

build:
  algorithm_path: /path/to/algorithm
  script_path: /path/to/algorithm/build.sh
```

| 选项名称 | 含义 | 备注 |
| --- | --- | --- |
| `algorithm` | 算法稳定标识 | 当前选择 `sfvision` 或 `vloc`，用于取得内置契约并关联同一算法的不同 commit 和 run |

配置格式版本、展示名称和内置契约版本由系统维护，不要求用户填写。系统使用 `algorithm` 查询算法注册表，从同一条注册记录取得后续运行和评估需要的全部固定信息。

#### 算法注册表

| 算法标识 | 展示名称 | 内置运行规则 | 固定输出 | voeval 工作流 | 当前可运行并评估的数据集类型 |
| --- | --- | --- | --- | --- | --- |
| sfvision | SFVision | sfvision 内置规则 | `vo.txt` | `sf_vo` | RK3399、RK3588 |
| vloc | VLOC | vloc 内置规则 | `vloc.txt`、`home_point.txt` | `sf_vloc` | RK3399、RK3588 |

每条内置运行规则绑定编译后的运行入口、参数映射和固定原始输出位置。Pipeline 根据数据集路径和 Segment 起止时间戳生成最终命令参数，直接执行算法，再把固定输出保存到当前数字 Segment 目录。新增算法时增加一条算法注册记录及其内置契约，不修改公共 Pipeline 或已有算法契约。

### 1. 算法编译配置接口

以 YAML 文档的形式配置，需要用户提供【组合编译命令所需的所有信息】。本系统中只有算法路径和编译脚本路径需要用户提供；具体编译操作由脚本内部负责，编译和运行环境默认已经准备完成。

| 选项名称 | 含义 | 备注 |
| --- | --- | --- |
| `build.algorithm_path` | 算法 Git 仓库根目录路径 | 系统在该目录中记录 Git、编译并运行算法 |
| `build.script_path` | 编译脚本文件路径 | 必须位于 `build.algorithm_path` 内 |

实际编译工作目录固定为 `build.algorithm_path`，Pipeline 直接执行 `build.script_path`，不创建算法源码副本。用户不再填写工作目录、构建命令、环境或产物路径。

编译执行、超时和成功判定统一在第四部分说明。

### 2. 算法运行配置接口

组合运行命令所需的信息都能由算法配置、内置契约和数据实例取得，因此用户 YAML 不再设置独立的 `run` 字段。信息来源如下：

| 信息 | 来源 |
| --- | --- |
| 算法目录和实际工作目录 | `build.algorithm_path` |
| 编译后的运行入口 | `algorithm` 对应的内置运行规则 |
| 数据集路径和 Segment 起止时间戳 | 数据实例及本次选择结果 |
| 参数映射和固定输出位置 | `algorithm` 对应的算法注册记录 |
| 数字 Segment 目录 | Pipeline 按本次 run 的稳定顺序从 0 开始自动创建 |
| 超时 | 系统全局默认 |
| 最终命令 | 运行模块自动生成 |

最终命令的生成、执行和成功判定统一在第五部分说明。Pipeline 直接执行命令参数列表，不生成或保存新的运行脚本。

### 3. 算法评估配置接口

模板要求提供【组合评估命令所需的所有信息】。这些信息都能由算法注册记录、数据实例和当前 Segment 运行结果获得，因此算法 YAML 不再设置独立的 `evaluation` 字段。

| 评估命令所需信息 | 来源 |
| --- | --- |
| ground truth 或参考数据所在目录 | 当前数据实例 |
| ground truth、标定和其他参考文件 | 数据集类型内置契约 |
| 算法估计结果所在目录 | 当前数字 Segment 目录 |
| 算法估计结果文件名 | 算法注册记录中的 `vo.txt` 或 `vloc.txt` |
| voeval 工作流 | sfvision 自动使用 `sf_vo`，vloc 自动使用 `sf_vloc` |
| 评估参数、指标和格式 | voeval 固定默认及系统全局设置 |

系统取得上述信息后，按照所选算法自动生成以下评估命令，用户不需要手动填写或修改命令：

```text
voeval sf_vo DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
voeval sf_vloc DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
```

其中 sfvision 使用第一条命令，vloc 使用第二条命令；数据目录、算法输出目录、数据集类型和指标保存路径均由 Pipeline 在本次 Segment 运行后自动代入。默认命令只保存指标 JSON，不生成单 Segment HTML 报告。固定输入格式、命令执行和成功判定统一在第六部分说明。

## 二、数据集插件配置

本章只定义用户首次录入一组数据集时需要填写的 YAML。用户提供数据集总根目录并选择数据集类型；文件布局、必需文件、分段方式和文件查找方式由系统中对应的数据集类型处理器提供。

需要区分三类信息：

1. **用户输入配置**：数据集总根目录和数据集类型；
2. **系统内置规则**：当前提供 RK3399、RK3588 和 KITTI Odometry 的识别与分段规则，并为 EuRoC 保留扩展位置；
3. **系统生成结果**：每个具体数据集自己的实例 YAML。

用户输入 YAML 保持为：

```yaml
dataset:
  root_path: /path/to/benchmark_datasets
  type: RK3399
```

### 1. 数据集路径

以 YAML 文件形式配置，需要用户提供【benchmark 测试的数据集总和的根目录】。该值必须是目录路径，不是目录名称，也不是某个 IMU、标定或真值文件的路径。

| 选项名称 | 含义 | 备注 |
| --- | --- | --- |
| `dataset.root_path` | benchmark 数据集总根目录路径 | 系统从该目录递归扫描多个同类型数据集 |

根目录规则：

- 同一次扫描不混合多种数据集类型；
- 普通运行用户不逐项提供 IMU、标定、真值或图像文件路径；
- 系统在每个候选数据集目录内按照所选类型自动寻找必需文件；
- 扫描和算法运行不得改写原始传感器数据；
- 每个数据实例 YAML 固定保存到该数据集自己的根目录，用户不能另行指定输出位置；
- 总根目录必须可读，每个具体数据集根目录必须可写，以便保存实例 YAML。

### 2. 数据集格式约定

以 YAML 文件形式配置，需要用户提供【确定当前数据集规则所需的信息】。对于当前已注册的内置类型，用户只需选择类型；该类型的必需文件、读取内容、分段方法和有效性判定来源均由系统内置。

| 选项名称 | 含义 | 备注 |
| --- | --- | --- |
| `dataset.type` | 本次扫描采用的数据集类型 | 当前可选 RK3399、RK3588 或 KITTI |

#### 数据集类型内置契约表

选择 `dataset.type` 后，Pipeline 自动加载对应的内置契约。下表中的内容全部由系统预设，不要求普通用户在数据集 YAML 中重复填写。

| 内置契约项 | 系统默认规则 | 用户是否填写 |
| --- | --- | --- |
| 类型标识 | 当前注册 RK3399、RK3588 和 KITTI Odometry；EuRoC 通过新增类型处理器接入 | 只选择类型 |
| 默认文件名与输出名 | 类型契约固定输入文件名；RK3399 使用单路 AVI，RK3588 使用四路 H.265，KITTI 使用官方双目 PNG 序列；数据实例配置固定命名为 `benchmark_dataset.yaml` 并保存到具体数据集根目录 | 否 |
| 必需文件 | RK3399 必须包含 `imu.txt`、`img.avi`、`imgts.txt`、`calib_raw.yaml`；RK3588 必须包含 `imu.txt`、四个 `video_*.h265`、两份时间戳和两份标定；KITTI 序列必须包含 `times.txt`、`calib.txt` 以及完整的 `image_0/image_1` 或 `image_2/image_3` 双目目录；KITTI `poses/序列号.txt` 是可选真值输入，数据集中的 `home_point.txt` 不作为 VLOC 输入 | 否 |
| 处理器读取的信息及用途 | RK3399、RK3588 读取时间戳和飞行状态；KITTI 读取时间戳并核对双目图像编号与数量；标定和可选真值只定位和校验，不转换原始数据 | 否 |
| 标定文件 | RK3399 校验 `calib_raw.yaml`；RK3588 校验前、下视两份标定；KITTI 根据实际采用的灰度或彩色双目校验 `calib.txt` 中对应的投影矩阵 | 否 |
| 分段依据 | RK3399、RK3588 使用 `flight_mode` 规则；KITTI 将 `times.txt` 的第一条到最后一条严格递增时间戳作为一个 Segment | 否 |
| 数据集有效性 | 所有必需文件存在且能够解析，并且至少形成一个有效 Segment | 否 |
| Segment 有效性 | 起止时间戳合法；RK3399 按 `imgts.txt` 计数，RK3588 按前、下视同步时间戳计数，KITTI 按与左右图像对一一对应的 `times.txt` 计数；输入图像帧不少于 200，且持续时间不少于 10 秒 | 否 |

缺失必需文件或解析失败时，系统拒绝录入并提示用户检查。能够解析但没有有效 Segment 时，系统仍保存各 Segment 及无效原因，将数据集标记为不可用并拒绝其进入 run。新增数据集类型时增加一份内置契约；同类型的新数据集直接复用已有契约。

RK3588 的 `video_bottom_0.h265`、`video_bottom_1.h265`、`video_front_0.h265`、`video_front_1.h265` 分别作为四路固定输入保存。`imgts_bottom.txt` 与 `imgts_front.txt` 分别解析并校验一致，一致后只形成一条同步时间序列，不能因双目和前后视角而重复计算帧数。

KITTI 在本设计中专指 KITTI Odometry，不包含 KITTI Raw 或 KITTI-360。每个 `sequences/XX` 目录独立录入；灰度和彩色双目同时存在时优先使用 `image_0/image_1`。`poses/XX.txt` 缺失或无效不会阻止算法运行，但实例会记录真值不可用，后续评估预检不得把该序列送入需要真值的评估链路。

具体数据集的实例 YAML 是扫描过程的系统输出，不属于数据集插件配置，其生成和保存要求统一在第三部分说明。

## 三、数据集管理

### 1. 数据集扫描与原信息保存

#### 1.1 输入

用户提供：

- 用户 YAML 中的 `dataset.root_path`；
- 用户 YAML 中的 `dataset.type`。

系统读取内置数据集类型契约及固定的最小输入图像帧数 200；该阈值不要求用户配置。

用户不提供额外规则文件或实例配置输出位置。系统根据数据集类型加载内置处理器，并优先读取每个数据集根目录中已有的实例 YAML：

- 实例 YAML 存在、能够读取、数据集类型一致且类型处理器版本未变化时，直接使用其中保存的 Segment，不重新读取原始数据；
- 实例 YAML 不存在或无法读取时，自动执行数据录入，重新识别 Segment 并把生成的实例 YAML 保存到该数据集根目录；
- 实例 YAML 中的数据集类型或类型处理器版本已经变化时，执行重新录入并更新实例 YAML；
- 系统不自动检测原始数据变化；用户修改数据后必须显式重新扫描。

算法失败阈值不属于数据集路径或数据集格式配置。系统默认值为 1，用户在选择本次运行的数据集或启动 run 时可以覆盖，最终值随本次 run 配置一起冻结。

#### 1.2 扫描过程

1. 规范化并检查数据集总根目录，确认目录存在且可读；
2. 加载本次唯一的数据集类型处理器；
3. 从总根目录递归扫描子目录；
4. 按该类型内置的目录和必需文件约定识别数据集候选目录；
5. 由类型处理器定位固定输入文件，并检查实际路径仍位于当前数据集目录范围内；
6. 在每个候选数据集根目录中查找系统固定名称的实例 YAML；
7. 实例 YAML 存在时读取其中的数据集身份、数据集类型、类型处理器版本和 Segment 列表；
8. 实例 YAML 内容完整、数据集类型一致且类型处理器版本未变化时，直接加载已保存的 Segment；
9. 实例 YAML 不存在、无法读取、内容不完整或类型不匹配时，确认该数据集根目录可写并进入数据录入流程；
10. 由类型处理器读取生成 Segment 所需的最小字段，并定位评估输入；
11. 按该类型内置的时间和分段方式提取全部 Segment；
12. 为每个 Segment 生成稳定标识，并保存原始起点时间戳、终点时间戳、持续时间和对应输入图像帧数；
13. 生成或更新该数据集的实例 YAML，并保存到该数据集根目录；
14. 根据每个 Segment 是否同时达到 200 个输入图像帧和 10 秒持续时间计算有效状态，汇总形成本次数据集列表。

同一个数据集内发现多个有效 Segment 时，每个 Segment 独立进入后续运行和评估。系统不把第一次开始到最后一次结束之间的内容合并为一个大区间。

未同时达到 200 个输入图像帧和 10 秒持续时间的 Segment 仍保存在对应数据集的实例 YAML 中，并在扫描结果中标记为无效，但不执行算法。没有有效 Segment 的数据集在当前扫描结果中标记为不可用；用户明确选择不可用数据集启动 run 时，系统在构建和运行前拒绝本次操作。

#### 1.3 输出

扫描至少产生以下结果：

| 输出 | 内容 | 使用者 |
| --- | --- | --- |
| 数据实例 YAML | 当前数据集的身份、类型处理器版本、实际文件路径，以及各 Segment 的稳定标识、起止时间戳、持续时间、输入图像帧数和有效状态 | 算法运行、评估和恢复 |
| 数据集列表 | 数据集名称、类型、有效 Segment 数和状态 | CLI 选择和报告 |
| 扫描诊断 | 未识别目录、缺失文件、解析失败和无效 Segment 原因 | 用户检查数据 |

建议的 CLI 交互为：

```text
benchmark dataset scan
benchmark dataset list
```

命令名称可以在实现时调整。`dataset scan` 同时承担首次数据录入和已有数据重新扫描，不再要求用户先执行格式注册。

用户启动 run 时，Pipeline 先读取所选数据集根目录下的实例 YAML，并直接取得已保存的 Segment。实例 YAML 缺失或无法读取时，系统自动调用上述数据录入流程；只有实例 YAML 成功生成并读取后，才允许进入构建和算法运行。算法运行阶段不再次解析原始文件来推导 Segment。

#### 1.4 更新与重新扫描

以下情况必须重新扫描数据集：

- 数据集总根目录发生变化；
- 用户修改了原始数据；
- 内置数据集类型处理器版本变化；
- 用户显式要求刷新数据集实例配置。

正常情况下，系统始终复用各数据集实例 YAML 中已经保存的 Segment，不重复分段。系统不自动检测原始数据变化，用户增加、删除或修改数据后必须显式重新扫描。固定帧数规则发生版本变化时通过提升类型处理器版本触发重新录入。历史 run 继续引用当时的数据实例和处理器版本快照。

## 四、算法编译模块

### 1. 编译命令组合

#### 1.1 输入

- 算法仓库根路径 `build.algorithm_path`；
- 编译脚本路径 `build.script_path`。

#### 1.2 处理过程

1. 校验算法插件和仓库路径；
2. 只读获取 Git HEAD、分支和工作区状态；
3. 确认编译脚本位于算法仓库内；
4. 固定使用 `build.algorithm_path` 作为工作目录；
5. 将 `build.script_path` 作为需要直接执行的编译命令；
6. 保存本次实际使用的算法路径、编译脚本路径和脚本内容摘要，不复制脚本正文。

本节保留模板要求的“编译命令组合”边界，但不生成构建计划，也不组合 CMake 参数或环境变量。需要的具体操作全部由编译脚本承担，系统只负责直接启动脚本并记录执行事实。

#### 1.3 输出

- 实际算法仓库路径；
- 实际编译脚本路径；
- 编译脚本内容摘要。

### 2. 编译算法

1. 以 `build.algorithm_path` 作为工作目录，直接启动 `build.script_path`；
2. 构建过程使用独立进程组；编译产物直接保留在算法目录中，Pipeline 不创建源码副本；
3. 实时保存标准输出和错误输出；
4. 进程结束后读取脚本退出码；
5. 退出码为 0 时记录编译脚本正常结束，非 0、超时或被中断时记录构建失败；
6. 脚本正常结束后，根据算法内置契约定位实际运行入口，并检查入口存在且可启动；已经存在且仍可用的编译产物可以直接使用；
7. 重新读取算法仓库的 Git 状态；
8. HEAD 或已跟踪源码在构建期间发生变化时，本次 run 停止继续执行。

构建阶段发生错误时，整次 run 不进入算法运行。构建错误属于配置或基础设施问题，不计入 Segment 的算法失败数量。

### 3. 编译结果回执确认

构建回执至少包含：

| 信息 | 含义 |
| --- | --- |
| `status` | success、failed、timeout 或 interrupted |
| `started_at`、`finished_at` | 构建开始和结束时间 |
| `duration_seconds` | 构建耗时 |
| `exit_code` | 构建进程退出码 |
| `stdout_path`、`stderr_path` | 构建日志路径 |
| `algorithm_path` | 本次实际编译和运行的算法仓库路径 |
| `script_path` | 实际执行的编译脚本路径 |
| `script_digest` | 编译脚本内容摘要，不保存脚本副本 |
| `resolved_entrypoint` | 按算法内置契约定位到的实际运行入口 |
| `failure_reason` | 构建失败原因 |

脚本退出码为 0 只表示编译脚本正常结束；只有实际运行入口也定位成功且可启动时，最终构建状态才为 success。脚本退出码非 0、超过系统全局超时或被中断时，构建失败。构建回执和日志立即写入本次 run，不等待整次测试结束后统一保存。

### 4. 编译代码版本记录

系统在构建前记录：

- Git 仓库根路径；
- 当前 HEAD commit；
- 当前分支或 detached HEAD 状态；
- 工作区是否存在未提交修改；
- 已跟踪源码的修改状态；
- 子模块版本；
- 所选算法标识及其注册记录摘要；
- 算法内置契约版本；
- 编译脚本路径及内容摘要。

构建后再次检查仓库状态。如果 HEAD 或已跟踪源码发生变化，系统停止继续执行并说明原因。由于系统直接使用算法仓库，run 进行期间用户不得修改源码、切换 commit、重新编译或替换运行入口。

Git 记录规则：

- 系统不执行 checkout、switch、pull、commit、reset 或 clean；
- dirty 工作区是否允许正式运行由策略控制，但必须显著记录；
- 同一 commit 可以产生多个 run，`test_id` 才是一次测试的唯一身份；
- 报告必须显示 commit 和 dirty 状态；
- 历史 run 保存自己的代码状态，不使用当前仓库状态重新解释。

## 五、算法运行模块

### 1. 运行命令组合

#### 1.1 输入

- 已在 `build.algorithm_path` 中完成编译并确认可用的运行入口；
- 已生成的数据实例配置；
- 当前有效 Segment；
- 用户选择的算法和数据集；
- 本次 run 的算法失败阈值，未设置时采用系统默认值 1；
- 系统为当前 Segment 分配的数字目录。

#### 1.2 处理过程

1. 使用 `algorithm` 查询算法注册表，取得运行入口、参数映射、固定输出和数据集允许范围，并检查当前数据集与 voeval 工作流兼容；
2. 使用第四部分已经确认的算法运行入口，不创建或定位算法副本；
3. 从当前数据实例 YAML 取得数据集路径和当前 Segment 的起止时间戳；
4. 在本次 run 的 `dataset/` 下为当前 Segment 创建独立的连续数字目录；
5. 根据算法内置契约，将运行入口、数据集路径和起止时间戳组合成最终命令参数列表；
6. 保存实际命令摘要，不生成或保存新的运行脚本。

每个有效 Segment 都单独生成命令并执行一次。两个 Segment 即使属于同一数据集，也不能共用输出文件或运行状态。

### 2. 执行运行命令

1. 为当前 Segment 创建独立进程组；
2. 使用最终参数列表启动算法，不交给登录 shell 二次解析；
3. 保存标准输出、错误输出、开始时间和进程信息；
4. 可选采集算法进程树的 CPU、内存和耗时；
5. 运行超时或用户终止时结束整个算法进程组；
6. 进程结束后，在算法契约绑定的固定位置检查输出：sfvision 为 `vo.txt`，vloc 必须同时包含 `vloc.txt` 和 `home_point.txt`，并确认文件由本次执行生成或更新；
7. 输出有效时，将全部算法输出保存到当前数字 Segment 目录；
8. 根据数据集类型只复制一份 voeval 使用的外参：RK3399 使用 `calib_raw.yaml`，RK3588 使用 `bottom_calib_raw.yaml`；不从数据集复制 `home_point.txt`；
9. 算法输出和外参保存成功后，将当前数字 Segment 目录作为 voeval 的 `log_dir` 进入评估模块；
10. 输出无效时不执行 voeval，将本 Segment 的评估状态记录为未执行；
11. 无论算法成功还是失败，都保存运行回执、评估状态和现有结果，并更新 `run_summary.xlsx`；
12. 当前数据集形成最终状态且 Excel 更新成功后提交数据集检查点，再开始下一个数据集。

算法失败处理：

- 进程失败、运行超时或必要输出无效属于算法失败；
- 每个 Segment 只执行一次，失败时计一次 Segment 失败；
- 默认采用“跳过并继续”模式：一个 Segment 失败后，将当前数据集标记为未运行成功，当前数据集剩余 Segment 标记为未运行，然后继续下一个数据集；
- 用户可以在启动时显式启用 `--fail-fast`：数据集预检、命令组合或算法运行首次失败时，保存当前事实和检查点后立即返回错误，不再启动后续数据集；
- 用户主动中断时两种模式都停止，不把人工中断当成算法失败；
- 本次 run 使用已经冻结的算法失败阈值；系统默认值为 1，用户可在选择数据集或启动 run 时覆盖；
- 默认模式即使运行过程中失败数量已经超过阈值，也继续处理后续数据集；
- 默认模式处理完所有数据集后，失败数量超过阈值才判定算法运行部分失败并生成失败的运行报告；`--fail-fast` 不等待失败阈值；
- 构建、数据配置、voeval 或存储问题不计入算法失败数量。

### 3. 运行结果回执检测

#### 3.1 成功判定

一个 Segment 运行成功必须同时满足：

1. 算法进程正常结束；
2. 退出码为 0；
3. 当前算法对应的固定输出文件存在且非空；
4. 输出满足算法内置契约规定的固定列数和基础格式；
5. 输出时间范围与当前 Segment 相容。

仅有退出码 0 不能判定运行成功。

固定输出缺失、为空、列数不足或基础格式错误属于算法输出失败，计入算法失败数量。VLOC 的 `vloc.txt` 或 `home_point.txt` 任一缺失都属于算法输出失败。通过检查后，Pipeline 将算法输出保存到当前数字 Segment 目录，并只从数据集复制当前类型对应的单份评估外参。外参复制失败属于存储问题，不计入算法失败；全部保存成功后才进入 voeval。格式合法但 voeval 无法计算指标属于评估失败，也不计入算法失败数量。

#### 3.2 Segment 运行回执

下表是 Pipeline 执行完成后自动生成的回执，不是用户需要填写的 YAML 配置。

| 信息 | 含义 |
| --- | --- |
| `test_id` | 本次完整测试身份 |
| `run_index` | 当前 Segment 在本次 run 中从 0 开始的连续编号 |
| `dataset_id` | 所属数据集 |
| `segment_id` | 当前独立运行单元 |
| `resolved_entrypoint` | 本次实际使用的算法运行入口 |
| `working_dir_path` | `build.algorithm_path` |
| `command` | 实际命令参数列表 |
| `started_at`、`finished_at` | 算法开始和结束时间 |
| `status` | success、failed、timeout 或 interrupted |
| `exit_code` | 算法退出码 |
| `stdout_path`、`stderr_path` | 运行日志路径 |
| `output_paths` | 实际输出绝对路径 |
| `output_checks` | 固定输出文件的检查结果 |
| `resource_usage` | 可选的算法进程资源统计 |
| `failure_reason` | 失败原因 |

#### 3.3 存储与检查点

一次 run 的结果目录按算法和 `test_id` 两层隔离。每个算法独立地从 `test-000` 开始递增，不按 commit 增加目录层级。系统仍读取并保存算法仓库当前 HEAD 的完整 commit；同一算法的任何历史测试都不得被后续测试覆盖。结构如下：

```text
result/
└── ALGORITHM_ID/
    └── TEST_ID/
        ├── config/
        ├── logs/
        │   ├── build.stdout.log
        │   └── build.stderr.log
        ├── build_receipt.yaml
        ├── checkpoint.yaml
        └── dataset/
            ├── 0/
            │   ├── receipt.yaml
            │   ├── stdout.log
            │   ├── stderr.log
            │   ├── FIXED_OUTPUT
            │   ├── CALIBRATION_FILE
            │   ├── home_point.txt          # 仅 sf_vloc
            │   └── evaluation/
            │       ├── metrics.json
            │       ├── receipt.yaml
            │       └── voeval.log
            ├── 1/
            └── ...
```

`ALGORITHM_ID` 就是算法名称，不拼接 hash。完整 commit、分支、dirty 状态和源码摘要保存在本次 `TEST_ID` 的冻结配置及构建回执中。全部有效 Segment 按稳定顺序从 0 开始预先编号；实际启动的 Segment 创建对应数字目录，被跳过的 Segment 不生成虚假目录，因此失败运行允许出现编号空缺。每份 Segment 回执仍保存原始 `dataset_id`、`segment_id` 和起止时间戳。数据集级运行状态集中写入 `checkpoint.yaml`，不再生成 `dataset_receipt.yaml`。

各阶段输出关系如下：

| 阶段 | 立即保存的内容 |
| --- | --- |
| 配置冻结 | 所选算法标识、算法注册记录及内置契约、数据集类型处理器版本、数据实例配置、固定最小输入图像帧规则、失败处理模式、算法失败阈值及其来源和本次参数快照 |
| Git 与构建 | 代码记录、算法仓库路径、编译脚本路径及内容摘要、退出码和日志 |
| Segment 运行 | 最终命令、运行回执、日志、算法固定输出和评估使用的单份外参；`sf_vloc` 的 `home_point.txt` 与轨迹一起由算法输出 |
| Segment 评估 | voeval 实际命令、日志、指标 JSON、状态和评估回执；默认不生成单项报告 |
| Segment 结束 | 按 run、数据集和 Segment 身份更新 `run_summary.xlsx`；发生评估时同时记录采用的 evaluation 身份 |
| 数据集结束 | 当前数据集的全部计划 Segment 形成最终状态后提交 run 检查点 |
| run 结束 | 完成 Excel 汇总和对比计算，只生成一份整次 run 的默认运行报告 |

历史事实只追加，不覆盖旧 run、旧 Segment 结果或旧 evaluation。

#### 3.4 暂停与恢复

- 用户请求暂停后，系统不再启动新数据集；当前数据集的全部计划 Segment 完成运行、评估、Excel 更新和检查点提交后再暂停；
- 检查点记录已经形成最终状态的数据集、下一个数据集位置、两种失败处理模式、失败阈值和已累计的算法失败数量；
- 重启后用户手动执行恢复；
- 恢复前重新检查 Git、所选算法标识及内置契约版本、算法路径、编译脚本路径及内容摘要、数据集类型处理器版本、数据实例配置、固定最小输入图像帧规则、本次 run 冻结的失败处理模式、算法失败阈值和 voeval 版本；
- 所有恢复条件相同才从检查点记录的未完成数据集继续；
- 检查点尚未提交的数据集从第一个有效 Segment 重新运行；恢复前只清理该数据集尚未提交的数字 Segment 目录；
- 已提交完成的数据集不重复运行，Excel 更新也不能产生重复行；
- 暂停期间保留已有算法结果、评估数据、Excel 和检查点，不生成最终运行报告；
- 上下文变化时拒绝使用旧检查点，要求创建新 run。

## 六、算法评估模块

### 1. 评估脚本运行

#### 1.1 触发条件

当前 Segment 的算法运行成功、输出检查通过且算法输出与单份外参已经保存到数字 Segment 目录后，Pipeline 立即发起 voeval 评估。评估不等待同一数据集或整次 run 的其他 Segment 完成。

#### 1.2 评估输入组合

1. 从当前 Segment 回执取得已经保存到数字 Segment 目录的算法估计输出；
2. 从当前数据实例配置取得包含 `imu.txt` 的数据集根目录；
3. 从 `algorithm` 对应的算法注册记录取得 `sf_vo` 或 `sf_vloc`；
4. 校验数据集根目录、当前数字 Segment 目录和 Segment 范围；
5. 直接将数据集根目录作为 voeval 的 `data_dir`，将已经包含算法输出和单份外参的当前数字 Segment 目录作为 `log_dir`；VLOC 的 `home_point.txt` 与 `vloc.txt` 均来自本次算法输出；
6. 按算法固定输出确认 VO 使用 `log_dir/vo.txt`，VLOC 使用 `log_dir/vloc.txt`，不再为评估重复复制这些文件；
7. 使用 CLI 默认的 `100 m` RPE 间隔调用 voeval；
8. 保存实际命令、voeval 版本、日志和指标 JSON，不请求生成 HTML 报告。

运行用户不填写参考文件名、估计轨迹文件名、轨迹格式或指标列表。Pipeline 按固定目录契约完成连接。

Pipeline 根据所选算法自动生成以下命令：

```text
voeval sf_vo DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
voeval sf_vloc DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
```

RK3399、RK3588 在命令中分别映射为 `rk3399`、`rk3588`。评估输入视图固定为：

| 工作流 | `data_dir` 固定内容 | `log_dir` 固定内容 |
| --- | --- | --- |
| `sf_vo` | `imu.txt` | `vo.txt` 和当前数据集类型对应的标定文件 |
| `sf_vloc` | `imu.txt` | `vloc.txt`、`home_point.txt` 和当前数据集类型对应的标定文件 |

评估视图中，RK3399 使用 `calib_raw.yaml`，RK3588 使用 `bottom_calib_raw.yaml`；RK3588 的 `front_calib_raw.yaml` 属于算法原始输入，不重复交给当前 voeval 工作流。固定解析器要求 IMU、VO、VLOC 文件分别至少包含前 21、11、13 列；允许尾部扩展列，缺少固定前缀列时评估失败。

当前数据录入处理器支持 RK3399、RK3588 和 KITTI Odometry；当前 voeval 工作流仍只支持 RK3399、RK3588。KITTI 可以完成录入和 Segment 生成，但在增加算法运行契约与对应评估入口前，不得进入现有 voeval 链路；EuRoC 暂时只有扩展接口。

#### 1.3 评估结果判定

| 情况 | 处理方式 | 是否计入算法失败 |
| --- | --- | --- |
| voeval 正常返回有效指标 | 保存成功结果 | 否 |
| 个别指标失效 | 保留失效状态和原因，其他指标仍可使用 | 否 |
| 输入已通过算法输出检查，但 voeval 无法完成指标计算 | 保存当前 Segment 的评估失败 | 否 |
| voeval 单次执行异常 | 保存日志和诊断，继续后续 Segment | 否 |
| voeval 整体不可用 | 在安全点暂停，修复后只重试评估 | 否 |

一次评估成功必须同时满足：voeval 退出码为 0，且指标 JSON 可读取。个别指标失效不等于整次评估失败，但必须保留失效状态和原因。

评估失败不重新运行已经成功的算法 Segment，也不删除算法原始输出。

### 2. 评估结果保存

#### 2.1 单 Segment 评估结果

每个 Segment 评估完成后只保存算法原始输出、voeval 通过 `-o` 生成的指标 JSON、评估日志和评估回执。默认流程不调用 `-s` 或 `--html-output`，因此不为单个 Segment 或单个数据集生成独立报告。用户需要查看单项轨迹细节时，可以基于已保存的算法输出另行调用 voeval，不影响 Pipeline 的默认运行报告。

Pipeline 不向 voeval 指标 JSON 中添加运行上下文字段，而是另外生成评估回执，用于关联 run、数据集、Segment 和对应文件。

每次评估生成独立 evaluation-id。Pipeline 评估回执至少保存：

| 信息 | 含义 |
| --- | --- |
| `evaluation_id` | 本次评估身份 |
| `test_id`、`dataset_id`、`segment_id` | 对应运行上下文 |
| `run_result_path` | 所使用的当前 Segment 算法输出路径 |
| `voeval_version` | 实际评估工具版本 |
| `evaluation_mode` | 本次采用的评估模式 |
| `voeval_arguments` | 实际工作流、数据集类型以及默认的 `delta=100`、`unit=m` |
| `input_digests` | 参考输入和估计输出摘要 |
| `voeval_metrics_path` | voeval 原始指标 JSON 路径 |
| `metrics` | 从原始 JSON 读取的指标摘要，不改变 voeval 指标含义 |
| `status` | success 或 failed |
| `failure_reason` | 评估失败原因 |

数值规则：

- JSON 不保存 NaN 或无穷值，使用空值加明确有效状态表示；
- 合法的零值必须保留；
- 失效值不参加百分比计算；
- 每个指标都显示单位和好坏方向；
- 指标定义变化后保留新版本，不能用同名指标静默覆盖旧含义。

#### 2.2 run Excel 数据汇总

Pipeline 为每个 run 维护一份 `run_summary.xlsx`。每个 Segment 形成最终状态后立即更新对应数据；暂停或重启时保留已经写入的内容，恢复后从检查点记录的未完成数据集重新开始。

Excel 只保存适合汇总和筛选的数据：

| 工作表 | 保存内容 |
| --- | --- |
| run 概要 | 算法、Git commit、dirty 状态、构建状态、失败阈值和 run 状态 |
| Segment 结果 | 数据集、Segment 起止时间戳、算法运行状态、评估状态、耗时和失败原因 |
| 指标结果 | 数据集、Segment、系统选定的报告关注指标、数值、单位、有效状态和失效原因 |
| 对比结果 | 参考 run、当前值、参考值、绝对变化、百分比变化和可比状态 |
| 结果索引 | run、数据集、Segment、采用的 evaluation-id，以及算法原始输出、全量指标 JSON、运行日志和评估回执的路径 |

voeval 的全量指标保存在每个 Segment 的 `metrics.json` 中，Excel 只提取系统选定的报告关注指标。算法轨迹、逐帧数据、完整日志和其他大文件内容不写入 Excel；结果索引只保存定位这些文件所需的路径。Excel 是整次 run 的累计数据汇总和最终报告的数据来源，不属于单 Segment 报告。

运行回执、评估回执和 `metrics.json` 是不可变的事实记录，Excel 是由这些记录生成的派生汇总，不是唯一事实源。Segment 状态按照 `test_id`、`dataset_id` 和 `segment_id` 幂等更新；存在评估结果时，再使用 `evaluation_id` 区分和选择评估记录，不能盲目追加重复行。当前数据集的 Excel 内容成功持久化后才能推进数据集检查点；Excel 损坏或缺失时可以从事实记录重建，run 完成后冻结本次默认汇总。

#### 2.3 run 汇总报告

Pipeline 在所有有效 Segment 形成最终状态、Excel 更新完成并完成参考结果对比后，只生成一份整次 run 的默认运行报告。即使算法失败数量超过阈值，系统也要先处理完剩余 Segment，再生成结果为失败的运行报告。

构建失败、算法运行入口无法定位、最终命令无法生成或其他不可恢复问题导致 run 提前终止时，系统根据已经保存的事实记录生成一份最小运行报告，说明终止阶段、原因和已有产物，不伪造未执行的 Segment 结果。可恢复的暂停不生成最终报告。

运行报告至少包含：

- 算法名称、Git commit 和 dirty 状态；
- 数据集总数、成功和失败数量及对应名单；
- Segment 总数、成功、失败和未执行数量及对应名单；
- 本次采用的算法失败阈值及其来源；
- 评估成功和评估失败的 Segment；
- 报告关注指标的汇总、变化趋势和失效数量；
- 失败 Segment 名单和原因；
- 与参考 run 或 baseline 的变化；
- `run_summary.xlsx`、算法原始结果和评估数据的保存位置；
- 本次配置、构建和结果目录。

Pipeline 负责生成整次 run 的唯一默认报告，voeval 在默认流程中只负责计算每个 Segment 的指标。暂停状态只保存已有结果、Excel 和检查点，不生成最终报告；恢复并完成全部 Segment 后再生成。

最终输出包括一份 `run_summary.xlsx` 和一份便于用户查看的运行报告。运行报告的最终载体可在实现时确认，CLI 必须输出 Excel 路径、报告路径和本次 run 结果根目录。

Excel 或运行报告生成失败属于汇总或存储问题，不计入算法失败数量。修复后只重新执行 Excel 重建、对比计算或报告生成，不重新构建和运行算法。

### 3. 评估基线对比

#### 3.1 可比性检查

只有满足以下条件的结果才能计算指标变化：

- `algorithm` 标识相同；
- 选择的数据集及有效 Segment 集合一致；
- 数据集类型、类型处理器版本和数据实例配置一致；
- Segment 起止范围一致；
- 实际运行选项和系统生成命令的语义一致；
- 指标名称、单位、方向和定义版本一致；
- voeval 模式、版本和实际 CLI 参数一致；
- 资源指标还要求机器和采集方式一致。

不同算法之间可以分别查看各自的 run 报告，但不进入“本次代码改进或退化”的回归结论。

#### 3.2 参考结果选择

参考结果支持：

1. **previous**：默认选择同一算法、不同 commit、最近一次兼容且成功的 run；
2. **baseline**：用户长期指定的同一算法成功 run；
3. **manual**：用户临时选择一个或多个兼容历史 run；
4. **none**：没有兼容参考时，最终运行报告只展示本次 run，不包含对比结论。

同一 commit 的重复 run 不自动作为 previous。用户可以手动选择它进行重复性查看，但不会改变默认参考，也不会自动设为 baseline。

临时比较不修改长期 baseline。用户选择多个历史 run 时，系统按输入顺序进行相邻逐版比较，也可以统一与 baseline 比较。

#### 3.3 指标变化计算

对每个相同数据集、相同 Segment、相同指标计算：

```text
absolute_change = current_value - reference_value
percent_change = absolute_change / abs(reference_value) × 100%
```

特殊情况：

- 参考值和当前值都为 0 时，百分比为 0；
- 参考值为 0、当前值非 0 时，不计算百分比，标注“从 0 变化”；
- 任一值失效时不计算变化，并记录失效原因；
- 参考成功而当前算法失败时，标注“可用性退化”，不把失败伪造成数值 0；
- 指标方向只用于说明增大或减小的含义，不自动替用户作回归判定。

#### 3.4 跨数据集汇总

不同数据集的 ATE、RPE 等原始值量级不同，系统不计算原始指标平均值。汇总方式为：

1. 先对每个可比 Segment 计算相对参考结果的百分比变化；
2. 再按同一指标汇总有效百分比变化；
3. 失效项不进入平均，但在结果旁标注“失效：数量”；
4. 同时保留每个数据集和 Segment 的明细，用户可以回到单项结果定位问题。

#### 3.5 对比输出

写入 Excel 和最终运行报告的对比结果至少包含：

- 当前 run 与每个参考 run 的身份；
- 可比或不可比结论及原因；
- 每个数据集、每个 Segment 的当前值、参考值和变化；
- 各指标有效百分比变化的汇总；
- 算法失败、评估失败和指标失效数量；
- baseline、previous 和手动参考的来源；
- 算法原始结果和单 Segment 评估数据路径。

默认流程不为每个 Segment 或每次对比单独生成报告；本次默认参考的对比结果写入 `run_summary.xlsx`，并统一进入整次 run 的唯一默认运行报告。用户手动重新选择参考结果属于独立的派生操作，可以在不重新构建、不重新运行算法、不重新计算原始指标的情况下生成新的对比数据；只有用户明确要求时才另行生成对比报告，且不覆盖默认运行报告或旧对比结果。
