# 算法基线测试系统 —— 详细设计文档（LLD）

> 文档状态：完整设计稿
>
> 编写依据：SRS、调研报告、HLD 及最新 `lld_template.md`
>
> 适用范围：本地单用户、CLI 手动启动的算法基线测试系统

本文严格保留模板规定的六个一级章节。存储、暂停恢复、异常处理、CLI 和报告等内容放入对应模块的小节中，不再增加新的一级章节。

本设计遵循以下已确认原则：

- 六个章节共同覆盖构建编译、代码版本记录、数据集管理、算法运行、结果评估、回归对比以及结果存储与报告；
- 本文具体描述 sfvision 和 vloc 两个算法，但公共流程不包含算法专用判断；
- 数据集类型处理器覆盖 RK3399、RK3588、KITTI 和 EuRoC；用户只选择类型，不填写文件匹配或 Segment 提取规则；
- 一次扫描只采用一种数据集类型，同一根目录下的数据按照该类型的内置规则识别；
- 每个有效 Segment 是独立的运行、评估和结果记录单元；每个数据集是检查点和异常恢复单元；
- 数据集参考文件由类型处理器自动发现并写入数据实例配置，voeval 工作流由所选算法的内置契约确定；
- Git 只用于读取和记录代码状态，系统不切换分支、不提交代码；
- 每个 Segment 运行结束后立即评估并保存算法原始输出和评估数据，选定数据写入测试任务级 Excel；单个 Segment 或数据集完成时不生成报告，整次测试任务结束后才生成汇总运行报告；
- 回归对比只用于同一算法的不同测试任务，不对不同算法给出“改进”或“退化”结论。

## 一、算法插件配置

每个算法首次接入时填写一份 YAML，以后直接复用。普通用户只选择一次算法，并提供算法路径、编译脚本路径和固定结果根目录；运行脚本生成规则、展示名称、固定输出和评估方式由系统内置的算法契约提供。

最小用户配置如下：

```yaml
algorithm: sfvision

build:
  algorithm_path: /path/to/algorithm
  script_path: /path/to/algorithm/build.sh

storage:
  result_root_path: /path/to/result
```


| 选项名称        | 含义     | 备注                                                       |
| ----------- | ------ | -------------------------------------------------------- |
| `algorithm` | 算法稳定标识 | 当前选择 `sfvision` 或 `vloc`，用于取得内置契约并关联同一算法的不同 commit 和测试任务 |
| `storage.result_root_path` | 固定结果根目录路径 | 同一算法的历史测试、baseline 关系和本次输出都保存在该目录下 |


配置格式版本、展示名称和内置契约版本由系统维护，不要求用户填写。三个用户提供的路径都按文件系统路径处理，加载后统一解析并保存为绝对路径。算法目录必须可读写，编译脚本必须可读且可执行，结果根目录必须可创建和写入。系统使用 `algorithm` 查询算法注册表，从同一条注册记录取得后续运行和评估需要的全部固定信息。

### 0. 系统内置算法契约

普通用户不填写运行入口、启动参数、算法资源路径、固定输出文件名或固定输出相对路径。这些信息由 Pipeline 为每个算法维护，避免同一信息在用户 YAML 中重复出现；用户仍需填写统一的结果根目录路径。

#### 算法注册表


| 算法标识     | 展示名称     | 运行脚本生成规则 | 固定输出相对路径  | voeval 工作流 | 当前可运行并评估的数据集类型 |
| -------- | -------- | -------- | --------- | ---------- | -------------- |
| sfvision | SFVision | sfvision 内置规则 | `vo.txt`  | `sf_vo`    | RK3399、RK3588  |
| vloc     | VLOC     | vloc 内置规则 | `vloc.txt` | `sf_vloc`  | RK3399、RK3588  |


每个算法的内置运行契约必须包含：


| 契约内容 | 在系统中的体现 | Pipeline 的处理 |
| --- | --- | --- |
| 编译产物位置 | 相对 `build.algorithm_path` 的运行入口定位规则 | 编译完成后解析为绝对路径并检查存在、可启动 |
| 启动参数映射 | 将数据集绝对路径、Segment 起止时间戳、工作目录和输出位置映射为该算法实际参数的模板 | 为每个 Segment 代入实际值并生成临时运行脚本 |
| 资源路径 | 算法运行所需模型、配置和其他资源相对算法仓库的位置 | 生成脚本时转换为绝对路径，避免依赖算法仓库作为当前工作目录 |
| 输出生成规则 | 固定输出名及其相对 Segment `work/` 的位置 | sfvision 定位 `vo.txt`，vloc 定位 `vloc.txt` |
| 输出校验规则 | 最小列数、非空要求和时间范围要求 | 决定算法输出是否有效以及是否进入 voeval |


固定输出路径相对当前 Segment 的 `work/` 工作目录解释，不相对算法仓库解释。每条内置运行规则负责从 `build.algorithm_path` 定位编译后的运行入口，使用绝对资源路径生成当前 Segment 的临时运行脚本。该脚本只在 `work/` 中执行，执行结束后删除。新增算法时只增加自己的算法注册记录和内置运行契约，不修改公共 Pipeline 或已有算法契约。

### 1. 算法编译配置接口

以 YAML 文档的形式配置，需要用户提供【组合编译命令所需的所有信息】。本系统中只有算法路径和编译脚本路径需要用户提供；具体编译操作由脚本内部负责，Pipeline 默认工作站环境已经配置完成。


| 选项名称                   | 含义             | 备注                            |
| ---------------------- | -------------- | ----------------------------- |
| `build.algorithm_path` | 算法 Git 仓库根目录路径 | 系统在该目录中编译算法，并从中定位编译后的运行入口 |
| `build.script_path`    | 编译脚本文件路径       | Pipeline 直接执行该路径指向的脚本 |


实际编译工作目录固定为 `build.algorithm_path`，Pipeline 直接执行 `build.script_path`，不创建算法源码副本。这里的算法目录只承担编译和定位编译产物，不作为后续算法运行阶段的系统工作目录；运行工作目录由 Pipeline 在 `storage.result_root_path` 的测试任务结果树下创建。用户不再填写工作目录、构建命令或产物路径。

编译执行、超时和成功判定统一在第四部分说明。

### 2. 算法运行配置接口

模板要求说明【组合运行命令所需的所有信息】。本系统不设置需要用户填写的 `run` 配置；编译完成后，Pipeline 根据算法内置契约和当前 Segment 自动生成临时运行脚本，并立即执行。这里的“临时运行脚本”只是把本次入口和参数展开成可执行命令的临时载体，不是用户维护的算法脚本，也不会修改算法仓库。

组合运行命令所需的信息来源如下：


| 信息 | 来源 |
| --- | --- |
| 算法源码和编译产物位置 | `build.algorithm_path` |
| 编译后的运行入口及启动方式 | `algorithm` 对应的内置运行脚本生成规则和 `build.algorithm_path` |
| 算法资源路径 | 内置规则规定模型、配置等资源相对 `build.algorithm_path` 的位置，运行前解析为绝对路径 |
| 系统运行工作目录 | Pipeline 在当前 Segment 结果目录下创建的 `work/` |
| 数据集路径和 Segment 起止时间戳 | 数据实例及本次选择结果 |
| 参数映射 | 内置规则把输入路径、起止时间戳、工作目录、输出位置和算法专用固定参数写入实际运行命令 |
| 固定输出源路径 | `algorithm` 对应的算法注册记录，相对 `work/` 解释 |
| 最终输出目录 | Pipeline 在当前 Segment 结果目录下创建的 `result/` |


运行脚本的生成、执行和成功判定统一在第五部分说明。临时运行脚本只服务于当前 Segment，生成后立即执行，执行结束后删除；用户不提供其路径，Pipeline 也不把它保存为测试任务结果文件。

### 3. 算法评估配置接口

模板要求提供【组合评估命令所需的所有信息】。这些信息都能由算法注册记录、数据实例和当前 Segment 运行结果获得，因此算法 YAML 不再设置独立的 `evaluation` 字段。


| 评估命令所需信息               | 来源                                        |
| ---------------------- | ----------------------------------------- |
| ground truth 或参考数据所在目录 | 当前数据实例                                    |
| ground truth、标定和其他参考文件 | 数据集类型内置契约                                 |
| 算法估计结果所在目录             | 当前 Segment 的 `result/` 目录                  |
| 算法估计结果文件名              | 算法注册记录中的 `vo.txt` 或 `vloc.txt`            |
| voeval 工作流             | sfvision 自动使用 `sf_vo`，vloc 自动使用 `sf_vloc` |
| 评估参数、指标和格式             | voeval 固定默认及系统全局设置                        |


系统取得上述信息后，按照所选算法自动生成以下评估命令，用户不需要手动填写或修改命令：

```text
voeval sf_vo DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
voeval sf_vloc DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
```

其中 sfvision 使用第一条命令，vloc 使用第二条命令；数据目录、算法输出目录、数据集类型和指标保存路径均由 Pipeline 在本次 Segment 运行后自动代入。默认命令只保存 voeval 的紧凑指标 JSON，不生成单 Segment HTML 报告。固定输入格式、命令执行和成功判定统一在第六部分说明。

## 二、数据集插件配置

本章只定义用户首次录入一组数据集时需要填写的 YAML。用户提供数据集总根目录并选择数据集类型；文件布局、必需文件、Segment 提取方式和文件查找方式由系统中对应的数据集类型处理器提供。

需要区分三类信息：

1. **用户输入配置**：数据集总根目录和数据集类型；
2. **系统内置规则**：RK3399、RK3588、KITTI、EuRoC 各自的数据识别、Segment 提取和有效性规则；
3. **系统生成结果**：每个具体数据集自己的实例 YAML。

用户输入 YAML 保持为：

```yaml
dataset:
  root_path: /path/to/benchmark_datasets
  type: RK3399
```

### 1. 数据集路径

以 YAML 文件形式配置，需要用户提供【benchmark 测试的数据集总和的根目录】。该值必须是目录路径，不是目录名称，也不是某个 IMU、标定或真值文件的路径。


| 选项名称                | 含义                  | 备注                 |
| ------------------- | ------------------- | ------------------ |
| `dataset.root_path` | benchmark 数据集总根目录路径 | 系统从该目录递归扫描多个同类型数据集 |


根目录规则：

- 同一次扫描不混合多种数据集类型；
- 普通运行用户不逐项提供 IMU、标定、真值或图像文件路径；
- 系统在每个候选数据集目录内按照所选类型自动寻找必需文件；
- 扫描和算法运行不得改写原始传感器数据；
- 每个数据实例 YAML 固定保存到该数据集自己的根目录，用户不能另行指定输出位置；
- 总根目录必须可读，每个具体数据集根目录必须可写，以便保存实例 YAML。

### 2. 数据集格式约定

以 YAML 文件形式配置，需要用户提供【确定当前数据集规则所需的信息】。对于当前已注册的内置类型，用户只需选择类型；该类型的必需文件、Segment 提取方法和有效性判定均由系统内置。


| 选项名称           | 含义           | 备注                          |
| -------------- | ------------ | --------------------------- |
| `dataset.type` | 本次扫描采用的数据集类型 | RK3399、RK3588、KITTI 或 EuRoC |


#### 数据集类型内置契约表

选择 `dataset.type` 后，Pipeline 自动加载对应的内置契约。下表中的内容全部由系统预设，不要求普通用户在数据集 YAML 中重复填写。


| 内置契约项 | 系统默认规则 | 用户是否填写 |
| --- | --- | --- |
| 类型标识 | 注册 RK3399、RK3588、KITTI 和 EuRoC；用户只选择其中一种 | 只选择类型 |
| 固定输入文件 | 类型契约绑定必需文件和标定文件；SF 数据默认包含 `imu.txt`、`img.avi`、`imgts.txt` 和对应标定文件，VLOC 评估还需要 `home_point.txt`；公开数据集采用其官方目录约定 | 否 |
| Segment 提取规则 | SF 数据只读取形成 Segment 所需的时间戳和 `flight_mode`；已预分段的公开数据集默认以完整序列作为一个 Segment | 否 |
| 数据集有效性 | 所有必需文件存在且能够读取，并且至少形成一个有效 Segment | 否 |
| Segment 有效性 | Segment 起止时间戳合法，且持续时间达到系统全局最小时长 | 否 |


缺失必需文件、Segment 提取失败或无法形成有效 Segment 时，系统拒绝录入整个有问题的数据集并提示用户检查。新增数据集类型时需要增加一份新的内置契约；同类型的新数据集直接复用已有契约。

具体数据集的实例 YAML 是扫描过程的系统输出，不属于数据集插件配置，其生成和保存要求统一在第三部分说明。

## 三、数据集管理

### 1. 数据集扫描与原信息保存

#### 1.1 输入

用户提供：

- 用户 YAML 中的 `dataset.root_path`；
- 用户 YAML 中的 `dataset.type`。

系统读取：

- 系统全局最小 Segment 时长。

用户不提供额外规则文件或实例配置输出位置。系统把实例配置固定命名为 `dataset.yaml`，保存在每个具体数据集根目录。系统根据数据集类型加载内置处理器，并优先读取已有的 `dataset.yaml`：

- 实例 YAML 存在、能够读取、完整通过固定结构校验且数据集类型一致时，直接使用其中保存的输入路径和 Segment，不重新读取原始数据来推导 Segment；
- 实例 YAML 不存在、无法读取或不符合固定结构时，自动执行数据录入，重新识别输入文件和 Segment，并把生成的实例 YAML 保存到该数据集根目录；
- 实例 YAML 中的数据集类型与当前选择不一致时，拒绝直接使用并提示用户显式刷新；
- 原始数据或内置 Segment 提取规则发生变化时，由用户显式刷新实例 YAML；系统不计算原始数据摘要来自动判断变化。

`dataset.yaml` 是 Pipeline 自动生成的内部配置，统一保存已解析出的绝对文件路径和 Segment。以 RK3399 为例：

```yaml
dataset:
  id: DATASET_ID
  type: RK3399
  root_path: /absolute/path/to/dataset

  input_paths:
    imu: /absolute/path/to/dataset/imu.txt
    image: /absolute/path/to/dataset/img.avi
    image_timestamps: /absolute/path/to/dataset/imgts.txt
    calibration: /absolute/path/to/dataset/calib_raw.yaml
    home_point: /absolute/path/to/dataset/home_point.txt

  segments:
    - id: SEGMENT_ID
      start_timestamp: 0
      end_timestamp: 0
      duration_seconds: 0
```

不适用于当前数据集类型的输入项不写入。`input_paths` 只保存类型处理器实际找到的文件，不保存文件匹配规则；`segments` 保存原始分段结果，是否达到本次全局最小时长在测试任务启动时重新计算。

算法失败阈值不属于数据集路径或数据集格式配置。系统默认值为 1，用户在选择本次运行的数据集或启动测试任务时可以覆盖，最终值随本次测试任务配置一起冻结。

#### 1.2 扫描过程

1. 规范化并检查数据集总根目录，确认目录存在且可读；
2. 加载本次唯一的数据集类型处理器；
3. 从总根目录递归扫描子目录；
4. 按该类型内置的目录和必需文件约定识别数据集候选目录；
5. 由类型处理器定位该数据集类型规定的固定输入文件；
6. 在数据集根目录中查找固定名称 `dataset.yaml`；
7. 对实例 YAML 校验固定结构：数据集身份、类型、根路径、必需输入绝对路径和 Segment 列表均存在，路径值确实是文件系统路径，每个 Segment 的身份、起止时间戳和时长完整且合法；
8. 实例 YAML 完整通过校验且数据集类型一致时，直接加载其中保存的输入路径和 Segment；
9. 实例 YAML 中的数据集类型与当前选择不一致时，拒绝使用并提示用户显式刷新；
10. 实例 YAML 不存在、无法读取或结构校验失败时，确认数据集根目录可写并进入数据录入流程；
11. 由类型处理器只读取提取 Segment 所需的文件和字段；
12. 按该类型内置的分段方式提取全部 Segment；
13. 为每个 Segment 生成稳定标识，并保存原始起点时间戳、终点时间戳和持续时间；
14. 生成或更新该数据集的实例 YAML，并保存到数据集根目录；
15. 根据实例 YAML 中的 Segment 和当前全局最小时长计算有效状态，汇总形成本次数据集列表。

同一个数据集内发现多个有效 Segment 时，每个 Segment 独立进入后续运行和评估。系统不把第一次开始到最后一次结束之间的内容合并为一个大区间。

未达到当前全局最小时长的 Segment 仍保存在实例 YAML 中，并在扫描结果中标记为无效，但不执行算法。没有有效 Segment 的数据集在当前扫描结果中标记为不可用；用户明确选择不可用数据集启动测试任务时，系统在构建和运行前拒绝本次操作。

#### 1.3 输出

扫描至少产生以下结果：


| 输出        | 内容                                                     | 使用者        |
| --------- | ------------------------------------------------------ | ---------- |
| 数据实例 YAML | 数据集身份、类型、根目录、已解析的必需输入绝对路径，以及各 Segment 的稳定标识、起止时间戳和持续时间 | 算法运行、评估和恢复 |
| 数据集列表     | 数据集名称、类型、有效 Segment 数和状态                               | CLI 选择和报告  |
| 扫描诊断      | 未识别目录、缺失文件、Segment 提取失败和无效 Segment 原因                    | 用户检查数据     |


建议的 CLI 交互为：

```text
benchmark dataset scan
benchmark dataset list
benchmark dataset show DATASET_ID
```

命令名称可以在实现时调整。`dataset scan` 同时承担首次数据录入和已有数据重新扫描，不再要求用户先执行格式注册。

用户启动测试任务时，Pipeline 先读取所选数据集的实例 YAML，并直接取得已保存的输入路径和 Segment。实例 YAML 缺失或无法读取时，系统自动调用上述数据录入流程。系统随后一次性预检所有选中数据集的必需输入、有效 Segment、算法支持范围和 voeval 支持范围；任一选中数据集不满足要求时，拒绝整个测试任务并集中列出问题，不进入编译。只有全部预检通过后，才允许进入构建和算法运行。算法运行阶段不再次读取原始文件来推导 Segment。

#### 1.4 更新与重新扫描

以下情况需要用户显式重新扫描数据集：

- 数据集总根目录发生变化；
- 原始数据发生变化；
- 数据集类型或内置 Segment 提取规则发生变化；
- 用户显式要求刷新数据集实例配置。

正常情况下，系统始终复用实例 YAML 中已经保存的 Segment，不重复分段。全局最小 Segment 时长变化时，也不需要重新读取原始数据；系统根据实例 YAML 中保存的原始 Segment 边界和时长重新计算当前运行视图。系统不自动检测原始数据变化；用户修改数据后必须显式刷新。历史测试任务继续引用当时的数据实例和全局阈值快照。

## 四、算法编译模块

### 1. 编译命令组合

#### 1.1 输入

- 算法仓库根路径 `build.algorithm_path`；
- 编译脚本路径 `build.script_path`。

#### 1.2 处理过程

1. 确认算法目录存在，编译脚本存在、可读取并具备执行权限；
2. 只读获取 Git HEAD、分支和工作区状态；
3. 固定使用 `build.algorithm_path` 作为编译工作目录；
4. 将 `build.script_path` 作为需要直接执行的编译命令；
5. 计算编译脚本内容摘要，并形成一份包含算法路径、编译脚本路径、脚本内容摘要和实际命令的构建摘要，作为 `build_receipt.yaml` 的初始内容；不复制脚本正文。

本节只保留模板要求的“编译命令组合”边界，不生成额外构建脚本或构建计划，也不组合 CMake 参数和产物规则。具体编译操作全部由用户提供的编译脚本承担。

#### 1.3 输出

构建开始前在 `storage.result_root_path/algorithms/ALGORITHM_ID/TEST_ID/build_receipt.yaml` 中写入构建摘要；构建完成后在同一回执中补充状态、退出码和日志路径，不另外创建重复的摘要文件，也不复制算法源码或脚本文件。

### 2. 编译算法

1. 以 `build.algorithm_path` 作为工作目录，直接启动 `build.script_path`；
2. 构建过程使用独立进程组；编译产生的文件直接保留在算法目录中，Pipeline 不复制或清理；
3. 实时保存标准输出和错误输出；
4. 进程结束后读取脚本退出码；
5. 退出码非 0、执行超时或被中断时记录编译失败；
6. 退出码为 0 时记录编译脚本正常结束；
7. 根据算法内置契约在 `build.algorithm_path` 中定位实际运行入口，解析为绝对路径并检查存在且可启动；预先已经存在且仍可用的编译产物可以直接被定位，不要求必须由本次脚本新建；
8. 记录实际运行入口路径；脚本退出码为 0 但入口无法定位或不可启动时，构建结果仍不能判定为成功；
9. 重新读取算法仓库的 Git HEAD 和已跟踪源码状态；
10. 构建期间 HEAD 或已跟踪源码发生变化时，本次测试任务停止继续执行。编译生成物不作为源码变化处理。

构建阶段发生错误时，整次测试任务不进入算法运行。构建错误属于配置或基础设施问题，不计入 Segment 的算法失败数量。

### 3. 编译结果回执确认

构建回执至少包含：


| 信息                          | 含义                                   |
| --------------------------- | ------------------------------------ |
| `status`                    | success、failed、timeout 或 interrupted |
| `started_at`、`finished_at`  | 构建开始和结束时间                            |
| `duration_seconds`          | 构建耗时                                 |
| `exit_code`                 | 构建进程退出码                              |
| `stdout_path`、`stderr_path` | 构建日志路径                               |
| `algorithm_path`            | 算法源码和编译产物所在的仓库路径                     |
| `script_path`               | 实际执行的编译脚本路径                          |
| `script_digest`             | 本次实际使用的编译脚本内容摘要，不保存脚本副本             |
| `resolved_entrypoint`       | 按算法内置契约定位到的实际编译产物或运行入口绝对路径           |
| `failure_reason`            | 构建失败原因                               |


脚本退出码非 0、执行超时或被中断时，构建失败。退出码为 0 只表示编译脚本正常结束；只有实际运行入口也定位成功且具备启动条件时，构建结果才为成功。即使构建成功，也不代表算法处理某个数据集一定成功，算法运行结果仍由第五部分判定。构建回执和日志立即写入本次测试任务，不等待整次测试结束后统一保存。

### 4. 编译代码版本记录

系统在构建前记录：

- Git 仓库根路径；
- 当前 HEAD commit；
- 当前分支或 detached HEAD 状态；
- 工作区是否存在未提交修改；
- 已跟踪源码的修改状态；
- 子模块版本；
- 所选算法标识及其注册记录摘要；
- 算法内置契约版本；
- 编译脚本路径、编译脚本内容摘要及算法内置运行脚本生成规则的摘要。

构建后再次检查仓库状态。如果 HEAD 或已跟踪源码发生变化，系统停止继续执行并说明原因。由于系统直接使用算法仓库，测试任务进行期间用户不得修改 HEAD、已跟踪源码、编译脚本或编译产物。

Git 记录规则：

- 系统不执行 checkout、switch、pull、commit、reset 或 clean；
- dirty 工作区允许运行，但必须在回执、Excel 和报告中显著记录；
- Pipeline 不监测未跟踪编译产物是否被外部替换；README 必须明确提示用户不得在测试任务进行期间重新编译或替换运行入口；
- 同一 commit 可以产生多个测试任务，`test_id` 才是一次完整测试的唯一身份；
- 报告必须显示 commit 和 dirty 状态；
- 历史测试任务保存自己的代码状态，不使用当前仓库状态重新解释。

## 五、算法运行模块

### 1. 运行命令组合

#### 1.1 输入

- 已完成编译的 `build.algorithm_path`；
- 所选算法的内置运行脚本生成规则；
- 系统已经生成并读取的数据实例 YAML；
- 当前有效 Segment；
- 用户选择的算法和数据集；
- 本次测试任务的算法失败阈值，未设置时采用系统默认值 1；
- 系统分配的当前 Segment 结果目录，以及其下的 `work/`、`result/` 和 `evaluations/`。

数据实例 YAML 是系统内部输入，用来取得已经解析的输入文件绝对路径和 Segment 起止时间戳，不要求用户在每次运行时填写。

#### 1.2 处理过程

1. 使用 `algorithm` 查询算法注册表，取得运行脚本生成规则、固定输出相对路径和数据集允许范围，并检查当前数据集与 voeval 工作流兼容；
2. 在当前 Segment 结果目录下创建独立的 `work/`、`result/` 和 `evaluations/`；
3. 固定使用当前 Segment 的 `work/` 作为系统运行工作目录；
4. 从数据实例 YAML 取得当前数据集根路径、必需输入绝对路径和当前 Segment 的起止时间戳；
5. 根据 `build.algorithm_path` 和算法内置契约定位编译后的运行入口及资源，并全部解析为绝对路径；
6. 由算法内置参数映射把运行入口、输入路径、起止时间戳、工作目录和输出位置组合成本 Segment 的临时运行脚本；
7. 检查生成结果完整，且所引用的运行入口和必需资源存在并具备使用条件；
8. 根据算法注册记录，在 `work/` 下确定固定输出源路径；
9. 将实际入口、资源路径、参数映射结果、输出源路径和最终命令摘要写入运行回执，但不保存临时运行脚本。

每个有效 Segment 都根据同一份算法内置规则重新生成自己的运行内容，并代入各自的数据集路径和起止时间戳后立即执行。两个 Segment 即使属于同一数据集，也不能共用工作目录、正式输出或运行状态。

### 2. 执行运行命令

1. 执行前确认算法仓库的 Git HEAD 和已跟踪源码与测试任务开始时一致；发生变化时停止本次测试任务，且不计为算法失败；
2. 记录算法固定输出源文件在运行前是否存在及其状态，防止误用上一 Segment 遗留的结果；
3. 为当前 Segment 创建独立进程组；
4. 在当前 Segment 的 `work/` 中直接执行系统刚生成的临时运行脚本；脚本通过绝对路径调用算法目录中的编译产物；
5. 保存标准输出、错误输出、开始时间和进程信息；
6. 可选采集算法进程树的 CPU、内存和耗时；
7. 运行超时或用户终止时结束整个算法进程组；
8. 进程结束后删除本 Segment 自动生成的临时运行脚本；无论执行成功、失败还是超时，都不把该脚本保存到结果目录；
9. 确认固定输出由本次执行新建或更新，并检查文件存在且非空；
10. 检查固定输出的列数、基础格式和时间范围；
11. 算法输出有效时，将固定输出复制到当前 Segment 的 `result/`；`result/` 中的文件是后续评估和历史追溯采用的正式输出，`work/` 中的源文件只作为诊断材料保留；
12. 正式输出保存成功后立即进入算法评估模块；
13. 算法输出无效时不执行 voeval，将本 Segment 的评估状态记录为未执行；
14. 固定输出复制或其他结果保存失败时，将 `persistence_status` 标记为失败，不执行本 Segment 评估，在 `test_summary.xlsx` 标出存储失败及原因，然后继续下一个 Segment；
15. 无论算法、评估或结果存储是否成功，都形成当前 Segment 的最终状态并幂等更新 `test_summary.xlsx`；
16. 当前数据集的全部计划 Segment 处理完后，若其最终状态已经持久化，则提交数据集检查点；若仍有待写内容，则保持检查点不变。没有暂停请求时继续下一个数据集并在后续安全点重试；已有用户暂停请求时不得启动新数据集。

算法失败处理：

- 算法进程已经成功启动后出现非零退出、被信号终止、运行超时或必要输出无效，属于算法失败；
- 失败统计以 Segment 为唯一计数单位；一个 Segment 无论出现多少个算法侧错误，都只计一次失败；
- 单个 Segment 失败后继续处理后续 Segment；
- 正常执行中不自动重跑失败 Segment；工作站异常中断后按数据集恢复属于唯一例外；
- 本次测试任务使用已经冻结的算法失败阈值；系统默认值为 1，用户可在选择数据集或启动测试任务时覆盖；
- 即使运行过程中失败数量已经超过阈值，也继续处理剩余 Segment；
- 所有 Segment 处理完后，只有 `失败 Segment 数 > 失败阈值` 时才判定本次测试任务的算法结果失败；等于阈值时仍判定为成功；
- 构建、数据配置、voeval 或存储问题不计入算法失败数量。

无法根据内置契约生成临时运行脚本、无法定位编译后的运行入口或临时脚本无法启动，属于算法接入契约或 Pipeline 问题，不计入 Segment 的算法失败数量；这类问题会终止本次测试任务，并保留已有结果和诊断。

状态逐层汇总如下：

- Segment 的算法状态由该次算法执行决定；
- 一个数据集的计划 Segment 全部完成时，若没有算法失败 Segment，则该数据集算法结果为成功；只要有一个算法失败 Segment，该数据集算法结果即为失败；
- 整次测试任务是否通过算法失败阈值，只统计所有数据集的失败 Segment 总数，不按失败数据集数统计；
- 测试任务执行完整性：所有计划 Segment 都形成最终状态时为完整，否则为未完整；
- 结果完整性：所有算法成功 Segment 的正式输出都保存成功时为完整；部分保存成功时为部分完整；没有可保存的算法成功 Segment 时记为不适用；
- 评估完整性：所有正式输出保存成功且应评估的 Segment 都有成功的默认评估时为完整；只有部分成功时为部分完整；没有应评估 Segment 时记为不适用；
- 测试任务执行完整性、算法结果是否通过阈值、结果完整性和评估完整性分别记录，不能用一个状态互相替代。

### 3. 运行结果回执检测

#### 3.1 算法运行成功判定

一个 Segment 运行成功必须同时满足：

1. 算法进程正常结束；
2. 退出码为 0；
3. 固定输出文件由本次执行新建或更新，并且存在且非空；
4. 输出满足算法内置契约规定的固定列数和基础格式；
5. 输出时间范围与当前 Segment 相容。

仅有退出码 0 不能判定运行成功。

固定输出没有在本次执行中生成或更新、缺失、为空、列数不足或基础格式错误，属于算法输出失败，计入算法失败数量。通过这些检查后，Pipeline 再复制结果并进入 voeval；结果复制、回执保存或检查点写入失败属于 Pipeline 存储问题，不计入算法失败。格式合法但 voeval 无法计算指标属于评估失败，也不计入算法失败数量。

#### 3.2 Segment 运行回执

下表是 Pipeline 执行完成后自动生成的回执，不是用户需要填写的 YAML 配置。


| 信息 | 含义 |
| --- | --- |
| `test_id` | 本次完整测试任务身份 |
| `dataset_id` | 所属数据集 |
| `segment_id` | 当前独立运行单元 |
| `contract_version` | 本次生成运行命令所采用的算法内置契约版本 |
| `resolved_entrypoint` | Pipeline 根据编译结果和内置契约定位到的实际运行入口 |
| `working_dir_path` | 实际系统运行工作目录，即当前 Segment 的 `work/` |
| `command` | 系统为当前 Segment 生成并实际执行的命令参数列表 |
| `resource_paths` | 本次使用的模型、配置和其他算法资源绝对路径 |
| `argument_mapping` | 数据集路径、起止时间戳、工作目录和输出位置对应到启动参数的实际结果 |
| `started_at`、`finished_at` | 算法开始和结束时间 |
| `algorithm_status` | success、failed、timeout 或 interrupted |
| `persistence_status` | 当前 Segment 所有应保存结果的总体存储状态；任一必需项失败时总体为失败 |
| `persistence_checks` | 固定输出、运行回执、日志、评估产物和 Excel 行各自的保存状态及失败原因 |
| `exit_code` | 算法退出码 |
| `stdout_path`、`stderr_path` | 运行日志路径 |
| `source_output_path` | 当前 Segment `work/` 下由算法契约规定的固定输出源路径 |
| `result_output_path` | 复制到当前 Segment `result/` 后的正式输出路径 |
| `output_checks` | 是否为本次新输出以及格式检查结果 |
| `resource_usage` | 可选的算法进程资源统计 |
| `failure_reason` | 算法失败或存储失败原因 |


#### 3.3 存储与检查点

Pipeline 使用 `storage.result_root_path` 指向的固定目录作为结果根目录。算法运行工作目录、正式输出目录和评估工作目录都创建在对应测试任务的结果树内；算法仓库只用于编译和定位可执行入口。系统在启动测试任务时自动生成不重复的 `test_id`，并创建固定的 `TEST_ID` 目录；同一 commit 多次测试也使用不同的 `TEST_ID`。`ALGORITHM_ID` 下不再增加 `runs/` 等中间层。建议结构如下：

```text
storage.result_root_path/
└── algorithms/
    └── ALGORITHM_ID/
        ├── history.yaml
        └── TEST_ID/
            ├── test.yaml
            ├── git.yaml
            ├── build_receipt.yaml
            ├── logs/
            │   ├── build.stdout.log
            │   └── build.stderr.log
            ├── datasets/
            │   └── DATASET_ID/
            │       ├── dataset.yaml
            │       └── segments/
            │           └── SEGMENT_ID/
            │               ├── work/
            │               ├── result/
            │               │   └── FIXED_OUTPUT
            │               ├── receipt.yaml
            │               ├── stdout.log
            │               ├── stderr.log
            │               └── evaluations/
            │                   └── EVALUATION_ID/
            │                       ├── work/
            │                       ├── metrics.json（成功时）
            │                       ├── receipt.yaml
            │                       └── voeval.log
            ├── checkpoint.yaml
            ├── test_summary.xlsx
            ├── comparisons/
            │   └── COMPARISON_ID/
            │       ├── comparison.yaml
            │       ├── comparison_result.xlsx
            │       └── comparison_report.<format>（按需）
            └── reports/
                └── test_report.<format>
```

`test.yaml` 是本次测试任务的统一冻结配置，合并保存用户选择、算法和构建路径、结果根目录、所选数据集实例路径、全局最小时长、失败阈值、内置契约版本和本次实际目录。每个测试任务内保存所用 `dataset.yaml` 的冻结副本；原数据集根目录中的实例文件继续作为以后运行的默认输入。

`history.yaml` 是系统维护的算法级历史配置，不由用户填写。它记录每个 `test_id` 的固定目录路径、commit、完成时间、执行完整性、算法结果、结果完整性和评估完整性，并保存用户指定 baseline 的 `test_id` 及固定目录路径。默认 previous 和 baseline 都从该文件解析，不依赖当前工作目录，也不每次遍历猜测历史目录。更新历史配置不得改写任何旧 `TEST_ID` 目录。

各阶段输出关系如下：


| 阶段         | 立即保存的内容                                                                            |
| ---------- | ---------------------------------------------------------------------------------- |
| 配置冻结       | `test.yaml`、所选数据集的 `dataset.yaml` 冻结副本、全局最小时长、算法失败阈值及其来源和本次参数快照               |
| Git 与构建    | 代码记录、算法仓库路径、编译脚本路径、实际命令、退出码和日志                                                   |
| Segment 运行 | `work/`、内置契约版本、实际运行入口、资源路径和参数映射、命令摘要、运行回执、日志，以及复制到 `result/` 的算法固定输出       |
| Segment 评估 | voeval 实际命令、日志、紧凑指标 JSON、状态和评估回执；默认不生成单项报告                                       |
| Segment 结束 | 按测试任务、数据集和 Segment 身份幂等更新 `test_summary.xlsx`；发生评估时同时记录采用的 `evaluation_id`               |
| 数据集结束     | 确认当前数据集全部计划 Segment 的最终状态已落盘，然后更新 `checkpoint.yaml`                                    |
| 测试任务结束 | 完成 Excel 汇总和对比计算，只生成一份整次测试任务的默认运行报告                                                |


已经提交数据集检查点的历史事实只追加，不覆盖旧测试任务、旧 Segment 结果或旧 evaluation。当前未提交数据集的内容不属于已完成事实，异常恢复时可以清理并重新生成。

#### 3.4 暂停与恢复

- `checkpoint.yaml` 只记录最后一个完整提交的数据集、已完成数据集顺序和对应汇总状态，不记录 Segment 内部执行阶段；
- 用户请求暂停后，系统不再启动新数据集；先处理完当前数据集的全部计划 Segment，并尝试保存 Excel 和提交数据集检查点，然后暂停；
- 当前数据集因待写内容无法提交新检查点时，暂停请求优先，系统在当前数据集边界进入暂停并保留上一个检查点；恢复时从上一个检查点之后重新执行整个未提交数据集；
- 重启后用户手动执行恢复；
- 恢复前重新检查 Git HEAD 和已跟踪源码、所选算法标识及内置契约版本、编译脚本路径和内容摘要、编译后实际运行入口、数据实例配置、全局最小时长、本次测试任务冻结的算法失败阈值和 voeval 版本；
- 所有恢复条件相同才允许使用原 `test_id` 恢复；
- 突然中断后，从检查点之后的第一个数据集重新开始；中断时正在处理的数据集从第一个计划 Segment 重新执行，不从 Segment 内部或该数据集中间继续；
- 已提交数据集不重复运行；未提交数据集目录中的半成品和 Excel 待提交行在恢复前按 `test_id`、`dataset_id` 清理或幂等替换；
- 因工作站异常中断而重新执行整个未提交数据集属于恢复，不算算法失败重跑；
- 暂停期间保留已有算法结果、评估数据、Excel 和检查点，不生成最终运行报告；
- 上下文变化时拒绝使用旧检查点，要求创建新测试任务。

固定输出或单个结果文件保存失败时，系统按上述规则标记当前 Segment 后继续运行，不计入算法失败数量。若 `test_summary.xlsx` 暂时无法写入，系统先在进程内保留待写行，在后续 Segment 和数据集结束时重试；系统仍继续处理后续数据集，但在所有此前待写行成功持久化前不得越过它们推进数据集检查点。若此时突然中断，恢复时从上一个已提交数据集之后重新开始，所有未被检查点覆盖的数据集都重新执行，不能把未落盘内容视为已完成。

## 六、算法评估模块

### 1. 评估脚本运行

#### 1.1 触发条件

当前 Segment 的算法运行成功、输出检查通过且固定输出已经复制到 `result/` 后，Pipeline 立即发起 voeval 评估。评估不等待同一数据集或整次测试任务的其他 Segment 完成。正式输出存储失败时不发起评估，只记录存储失败并继续后续 Segment。

#### 1.2 评估输入组合

1. 从当前 Segment 回执取得已经复制到 `result/` 的算法估计输出；
2. 从本次测试任务冻结的 `dataset.yaml` 取得 `input_paths` 中的 IMU、标定和其他参考输入绝对路径；
3. 从 `algorithm` 对应的算法注册记录取得 `sf_vo` 或 `sf_vloc`；
4. 校验输入文件存在、可读取并且与当前 Segment 范围相容；
5. 在当前 Segment 的 `evaluations/EVALUATION_ID/work/` 中组成固定的 `data_dir` 和 `log_dir`；
6. 将估计输出放入固定位置：VO 使用 `log_dir/vo.txt`，VLOC 使用 `log_dir/vloc.txt`；
7. 使用 CLI 默认的 `100 m` RPE 间隔调用 voeval；
8. 保存实际命令、voeval 版本、日志和评估回执；voeval 成功写出紧凑指标 JSON 时再保存 `metrics.json`，不请求生成 HTML 报告。

运行用户不填写参考文件名、估计轨迹文件名、轨迹格式或指标列表。Pipeline 按固定目录契约完成连接。

Pipeline 根据所选算法自动生成以下命令：

```text
voeval sf_vo DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
voeval sf_vloc DATA_DIR LOG_DIR --dataset DATASET_TYPE -o METRICS_PATH
```

RK3399、RK3588 在命令中分别映射为 `rk3399`、`rk3588`。评估输入视图固定为：


| 工作流       | `data_dir` 固定内容 | `log_dir` 固定内容                              |
| --------- | --------------- | ------------------------------------------- |
| `sf_vo`   | `imu.txt`       | `vo.txt` 和当前数据集类型对应的标定文件                    |
| `sf_vloc` | `imu.txt`       | `vloc.txt`、`home_point.txt` 和当前数据集类型对应的标定文件 |


RK3399 的标定文件为 `calib_raw.yaml`，RK3588 为 `bottom_calib_raw.yaml`。固定解析器要求 IMU、VO、VLOC 文件分别至少包含前 21、11、13 列；允许尾部扩展列，缺少固定前缀列时评估失败。

当前 voeval 工作流只直接支持 RK3399 和 RK3588。KITTI、EuRoC 可以完成数据录入，但在增加对应的输入转换或 voeval 工作流前，运行预检必须拒绝进入当前评估链路。

#### 1.3 评估结果判定


| 情况                            | 处理方式                 | 是否计入算法失败 |
| ----------------------------- | -------------------- | -------- |
| voeval 正常返回有效指标               | 保存成功结果               | 否        |
| 个别指标失效                        | 保留失效状态和原因，其他指标仍可使用   | 否        |
| 输入已通过算法输出检查，但 voeval 无法完成指标计算 | 保存当前 Segment 的评估失败   | 否        |
| voeval 单次执行异常                 | 保存日志和诊断，继续后续 Segment | 否        |
| voeval 整体不可用                  | 标记评估失败并继续保存后续算法输出    | 否        |


一次评估成功必须同时满足：voeval 退出码为 0，指标 JSON 可读取，并且包含当前工作流预期的顶层结构。个别指标为空不等于整次评估失败，但必须由 Pipeline 标记为失效，并记录“字段缺失”或“voeval 返回空值”；Pipeline 不根据日志猜测数值。voeval 进程或输入级失败原因保存在评估回执和日志中。

评估失败不重新运行已经成功的算法 Segment，也不删除算法原始输出。

### 2. 评估结果保存

#### 2.1 单 Segment 评估结果

每个 Segment 评估完成后保存算法原始输出、评估日志和评估回执；评估成功时还保存 voeval 通过 `-o` 生成的紧凑指标 JSON。默认流程不调用 `-s` 或 `--html-output`，因此不为单个 Segment 或单个数据集生成独立报告。用户需要查看单项轨迹细节时，可以基于已保存的算法输出另行调用 voeval，不影响 Pipeline 的默认运行报告。

Pipeline 不向 voeval 指标 JSON 中添加运行上下文字段，而是另外生成评估回执，用于关联测试任务、数据集、Segment 和对应文件。

voeval 当前由 `-o` 写出的不是完整评估报告，而是紧凑指标结果：

| 工作流 | 紧凑 JSON 的固定内容 |
| --- | --- |
| 公共部分 | `mode`；`ate_position_m` 的 rmse、mean、median、max、min；`rpe_translation_m` 的相同统计量及 `delta_value`、`delta_unit`、`count`；以及 `segment_count` |
| `sf_vo` | 公共部分，加 `sim3` 的连续评估段数量，以及各连续评估段的 scale、rotation、translation |
| `sf_vloc` | 公共部分，加 `vloc_metrics` 中的轨迹长度、水平/垂直位置误差和姿态误差摘要 |

这里的 `segment_count` 和 `sim3.segments` 表示 voeval 在当前一次评估内根据 reset 或断点识别出的连续评估段，不是 Pipeline 在 `dataset.yaml` 中登记并用于失败计数的 Segment。Pipeline 维护固定指标目录，为报告所采用的标量指标补充稳定名称、单位、好坏方向和有效性规则；这些元信息写入评估回执、Excel 和报告，不反向修改 voeval 原始 JSON。Sim3 的矩阵和向量完整保存在原始 JSON 中，只将适合比较的标量写入 Excel。

每次评估生成独立 `evaluation_id`。Pipeline 评估回执至少保存：


| 信息                                 | 含义                                    |
| ---------------------------------- | ------------------------------------- |
| `evaluation_id`                    | 本次评估身份                                |
| `test_id`、`dataset_id`、`segment_id` | 对应测试任务和运行上下文                         |
| `algorithm_result_path`            | 所使用的当前 Segment 算法输出路径                 |
| `voeval_version`                   | 实际评估工具版本                              |
| `evaluation_mode`                  | 本次采用的评估模式                             |
| `voeval_arguments`                 | 实际工作流、数据集类型以及默认的 `delta=100`、`unit=m` |
| `input_paths`                      | 本次使用的参考输入和估计输出路径                     |
| `voeval_metrics_path`              | voeval 成功时生成的原始紧凑指标 JSON 路径；失败时为空    |
| `metrics`                          | 从紧凑 JSON 读取的报告指标摘要，不改变 voeval 指标含义   |
| `status`                           | success 或 failed                      |
| `failure_reason`                   | 评估失败原因                                |


数值规则：

- voeval 原始 JSON 把 NaN 或无穷值写成空值，但不附加有效状态字段；Pipeline 在评估回执、Excel 和报告中另外保存有效状态和失效原因；
- 合法的零值必须保留；
- 失效值不参加百分比计算；
- 每个指标都显示单位和好坏方向；
- 指标定义变化后保留新版本，不能用同名指标静默覆盖旧含义。

同一 Segment 可以因用户独立重试评估而产生多个 `evaluation_id`，所有尝试均保留且互不覆盖。在同一冻结评估契约下，默认采用最近一次成功的评估；没有成功评估时，采用最近一次失败记录表达状态。测试任务完成后，默认 Excel 和报告冻结；之后新增的独立评估不覆盖原汇总，只能用于新的派生对比或用户单独查看。

#### 2.2 测试任务 Excel 数据汇总

Pipeline 为每个测试任务维护一份 `test_summary.xlsx`。每个 Segment 形成最终状态后立即幂等更新对应数据；结果文件存储失败也必须形成一行并标出失败位置和原因。突然中断后按数据集检查点恢复，未提交数据集的旧行由同一 `test_id`、`dataset_id` 和 `segment_id` 幂等替换，不重复追加。

Excel 只保存适合汇总和筛选的数据：


| 工作表        | 保存内容                                                              |
| ---------- | ----------------------------------------------------------------- |
| 测试概要       | 算法、Git commit、dirty 状态、构建状态、失败阈值和测试任务状态                         |
| Segment 结果 | 数据集、Segment 起止时间戳、算法运行状态、总体存储状态及各存储分项、评估状态、耗时和失败原因                 |
| 指标结果       | 数据集、Segment、系统选定的报告关注指标、数值、单位、有效状态和失效原因                           |
| 对比结果       | 参考测试任务、当前值、参考值、绝对变化、百分比变化和可比状态                                  |
| 结果索引       | 测试任务、数据集、Segment、采用的 `evaluation_id`，以及算法原始输出、紧凑指标 JSON、运行日志和评估回执的路径 |


voeval 成功时生成的紧凑指标保存在对应 `evaluation_id` 目录的 `metrics.json` 中，Excel 只提取系统固定指标目录选定的报告关注指标。算法轨迹、逐帧数据、完整日志和其他大文件内容不写入 Excel；结果索引只保存定位这些文件所需的路径。Excel 是整次测试任务的累计数据汇总和最终报告的数据来源，不属于单 Segment 报告。

已提交数据集的运行回执、评估回执和成功生成的 `metrics.json` 是不可变事实记录，Excel 是由这些记录生成的派生汇总，不是唯一事实源。Segment 状态按照 `test_id`、`dataset_id` 和 `segment_id` 幂等更新；存在评估结果时，再使用 `evaluation_id` 区分和选择评估记录，不能盲目追加重复行。Excel 成功持久化后才能提交对应数据集检查点；Excel 损坏或缺失时可以从事实记录重建，测试任务完成后冻结本次默认汇总。

#### 2.3 测试任务汇总报告

Pipeline 在所有有效 Segment 形成最终状态、Excel 更新完成并完成参考结果对比后，只生成一份整次测试任务的默认运行报告。即使算法失败数量超过阈值，系统也要先处理完剩余 Segment，再生成结果为失败的运行报告。

构建失败、临时运行脚本无法生成或无法启动，以及其他不可恢复问题导致测试任务提前终止时，系统根据已经保存的事实记录生成一份最小运行报告，说明终止阶段、原因和已有产物，不伪造未执行的 Segment 结果。可恢复的暂停不生成最终报告。

运行报告至少包含：

- 算法名称、Git commit 和 dirty 状态；
- 测试任务执行完整性、算法结果、结果完整性和评估完整性；
- 数据集总数、算法成功和算法失败数量及对应名单；数据集内任一 Segment 算法失败时，该数据集计为算法失败；
- Segment 总数、成功、失败和未执行数量及对应名单；
- 本次采用的算法失败阈值及其来源、失败 Segment 总数，以及总数是否超过阈值；
- 评估成功和评估失败的 Segment；
- 结果存储成功和存储失败的 Segment 及原因；
- 报告关注指标的汇总、变化趋势和失效数量；
- 失败 Segment 名单和原因；
- 与参考测试任务或 baseline 的变化；
- `test_summary.xlsx`、算法原始结果和评估数据的保存位置；
- 本次配置、构建和结果目录。

Pipeline 负责生成整次测试任务的唯一默认报告，voeval 在默认流程中只负责计算每个 Segment 的指标。暂停状态只保存已有结果、Excel 和检查点，不生成最终报告；恢复并完成全部 Segment 后再生成。

最终输出包括一份 `test_summary.xlsx` 和一份便于用户查看的运行报告。运行报告的最终载体可在实现时确认，CLI 必须输出 Excel 路径、报告路径和本次测试任务结果根目录。

Excel 或运行报告生成失败属于汇总或存储问题，不计入算法失败数量。修复后只重新执行 Excel 重建、对比计算或报告生成，不重新构建和运行已经提交检查点的数据集；未提交数据集仍按第五部分的检查点规则重新执行。

### 3. 评估基线对比

#### 3.1 可比性检查

只有满足以下条件的结果才能计算指标变化：

- `algorithm` 标识相同；
- 选择的数据集及有效 Segment 集合一致；
- 数据集标识、数据集类型和本次采用的数据实例配置一致；
- Segment 起止范围一致；
- 算法内置契约版本一致；
- 实际运行选项和系统生成命令的语义一致；
- 指标名称、单位、方向和定义版本一致；
- voeval 模式、版本和实际 CLI 参数一致；
- 资源指标还要求机器和采集方式一致。

可比性按具体指标对检查，不要求失败或失效项伪造数值。当前或参考 Segment 缺少有效指标时，该项标记为不可比；参考成功而当前算法失败时，另行标记为可用性退化。

不同算法之间可以分别查看各自的测试报告，但不进入“本次代码改进或退化”的回归结论。

#### 3.2 参考结果选择

参考结果支持：

1. **previous**：从算法级 `history.yaml` 中默认选择同一算法、不同 commit、最近一次兼容且具备默认参考资格的测试任务；
2. **baseline**：用户在 `history.yaml` 中长期指定的同一算法兼容测试任务及其固定目录路径；该任务必须具备默认参考资格；
3. **manual**：用户临时选择一个或多个兼容历史测试任务；
4. **none**：没有兼容参考时，最终运行报告只展示本次测试任务，不包含对比结论。

同一 commit 的重复测试任务不自动作为 previous。用户可以手动选择它进行重复性查看，但不会改变默认参考，也不会自动设为 baseline。

默认参考资格同时要求：测试任务执行完整、失败 Segment 数未超过该任务自己的阈值、结果完整性为完整、评估完整性为完整，并且至少存在一个有效的报告关注指标。测试任务可以在阈值范围内包含少量算法失败 Segment，仍可作为 previous 或 baseline；实际比较时只计算双方均有有效值的指标，并把缺失结果和可用性退化单独展示。临时比较不修改长期 baseline。用户选择多个历史测试任务时，系统按输入顺序进行相邻逐版比较，也可以统一与 baseline 比较。

#### 3.3 指标变化计算

对每个相同数据集、相同 Segment、相同指标计算：

```text
absolute_change = current_value - reference_value
percent_change = absolute_change / abs(reference_value) × 100%
```

特殊情况：

- 参考值和当前值都为 0 时，百分比为 0；
- 参考值为 0、当前值非 0 时，不计算百分比，标注“从 0 变化”；
- 任一值失效时不计算变化，并记录失效原因；
- 参考成功而当前算法失败时，标注“可用性退化”，不把失败伪造成数值 0；
- 指标方向只用于说明增大或减小的含义，不自动替用户作回归判定。

#### 3.4 跨数据集汇总

不同数据集的 ATE、RPE 等原始值量级不同，系统不计算原始指标平均值。汇总方式为：

1. 先对每个可比 Segment 计算相对参考结果的百分比变化；
2. 同一数据集包含多个可比 Segment 时，先计算该数据集内部的有效百分比变化平均值；
3. 再让各数据集等权，汇总同一指标的有效百分比变化，避免 Segment 较多的数据集获得更高权重；
4. 失效项不进入平均，但在结果旁标注“失效：数量”；
5. 同时保留每个数据集和 Segment 的明细，用户可以回到单项结果定位问题。

#### 3.5 对比输出

写入 Excel 和最终运行报告的对比结果至少包含：

- 当前测试任务与每个参考测试任务的身份；
- 可比或不可比结论及原因；
- 每个数据集、每个 Segment 的当前值、参考值和变化；
- 各指标有效百分比变化的汇总；
- 算法失败、评估失败和指标失效数量；
- baseline、previous 和手动参考的来源；
- 算法原始结果和单 Segment 评估数据路径。
