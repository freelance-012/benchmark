# 算法基线测试系统 —— 需求规格说明（SRS）

## 1. 背景与目标

当前算法基线测试流程不统一，需要人工编译，人工运行测试数据，人工评估算法性能，效率很低，每个数据都要人工手动运行、评估，没法大批量测试。而且，当前基线测试数据并不固定，评估结果并不保存，没法与历史版本进行对比分析。

**痛点：** 效率很低、流程不固定

现在亟需一个基线测试系统，在固定数据集上自动化完成算法构建编译、测试运行、性能评估，并保存评估结果，用于对比分析；

**成功的标志：** 对于一个新的定位算法，基线测试系统进行简单配置之后，即可自动完成对该算法的构建编译、测试运行、性能评估，并保存评估结果。

## 2. 用户与使用场景

用户是定位算法研发人员，在本地使用，终端运行。

典型使用场景：

1. 用户可以提供当前算法的构建方式、运行方式（所需参数）、轨迹文件日志目录、轨迹评估方式、所需测试数据集路径等信息，供系统配置；
2. 用户可以运行基线测试运行命令。系统会自动记录当前算法的git仓库状态，当前HEAD，开始自动构建并编译当前算法，生成可执行文件。并依次运行数据集中的每个测试数据，并对每个数据进行性能评估，最终输出报告；
3. 用户可以查看报告，报告需要记录每个数据集上算法的性能指标。
4. 用户可以用系统进行性能对比。 以本轮测试结果和基线测试结果的指标进行对比，得到对比报告。

## 3. 功能需求

| 编号 | 功能 | 描述 |
| --- | --- | --- |
| F1 | 构建编译 | 用户发起基线测试后，系统应根据已接入算法的构建要求完成构建，并明确给出算法是否具备运行条件。 |
| F2 | 记录代码版本 | 系统应记录本次测试所对应的代码版本及代码仓库状态，使测试结果能够追溯到实际参与测试的代码。 |
| F3 | 数据集管理 | 系统应允许用户选择本次测试使用的本地数据集，确认数据集是否满足测试要求，并保存本次测试与数据集之间的对应关系。 |
| F4 | 运行算法 | 系统应按照本次测试范围运行算法，使每个测试数据都得到独立的运行结果，并记录运行成功、失败及必要的过程信息。 |
| F5 | 评估结果 | 系统应对满足评估条件的算法输出进行性能评估，得到每个测试数据对应的性能指标和评估结论。 |
| F6 | 回归对比 | 系统应支持将本次评估结果与基线结果或历史结果进行对比，展示各项指标的变化，帮助用户判断算法修改产生的影响。 |
| F7 | 结果存储、可视化与报告 | 系统应保存每次测试的代码版本、数据集、运行结果、评估结果和对比结果，并形成便于查看的可视化和报告。 |

## 4. 核心难点：算法可扩展性

系统需要支持一至两个指定的定位算法，并具备接入其他定位算法的能力。不同算法在构建、运行、输出、评估和结果展示方面存在差异，系统必须允许这些差异存在，不能要求所有算法采用完全相同的内部实现。

不同算法的差异至少包括：

1. 构建所需的环境、依赖、步骤和成功条件不同；
2. 运行时需要的数据、参数、资源和启动条件不同；
3. 算法产生的输出内容、格式和成功判定方式不同；
4. 可使用的评估方式、评估指标和指标含义不同；
5. 需要展示的可视化内容和报告重点不同。

新增一个算法时，算法提供者应提供以下信息：

1. 算法的名称、用途和适用范围；
2. 算法完成构建所需的条件及构建成功标准；
3. 算法运行所需的输入、参数、环境及运行成功标准；
4. 算法适用的数据集要求；
5. 算法输出的内容、含义和有效性判断标准；
6. 算法结果对应的评估方式、关注指标和展示要求；
7. 用于确认算法已经正确接入系统的测试条件。

系统对算法扩展应满足以下要求：

- 新增算法时，只需要补充该算法自身的接入信息，不应要求修改已有算法的接入内容；
- 新增算法不应改变构建、运行、评估、对比、存储和报告等公共业务目标；
- 一个算法的接入或运行问题不应影响其他已接入算法及其历史结果；
- 算法正式参与基线测试前，应能够确认其构建、运行、输出和评估要求均已满足；
- 已保存的历史结果应继续保持原有含义，不因增加新算法而失效。

## 5. 职责边界

### 5.1 本系统负责

- 接收并管理算法完成基线测试所必需的信息；
- 记录本次测试对应的代码版本和测试范围；
- 组织算法构建、数据集运行和结果评估；
- 保存每次测试已经产生的运行结果和评估结果；
- 对可比较的当前结果、基线结果和历史结果进行回归对比；
- 生成面向算法研发人员的可视化和测试报告；
- 支持接入新的定位算法。

### 5.2 本系统不负责（明确排除）

- 不负责修改定位算法的功能代码或替算法研发人员修复算法问题；
- 不负责替算法研发人员定义缺失的构建要求、运行要求、输出含义或评估标准；
- 不负责生产、修复或标注测试数据集，测试数据及其必要说明由用户提供；
- 本项目不负责监听代码提交或自动触发测试，也不包含 CI/CD 集成、Web 管理、多用户协作、多机执行或分布式调度；
- 不负责管理 Git 分支、切换代码版本或向代码仓库提交修改。

### 5.3 与外部系统的边界

| 外部对象 | 职责边界 |
| --- | --- |
| voeval | voeval 负责根据算法输出计算评估指标；本系统负责发起评估、关联测试上下文、保存评估结果并用于汇总和对比。 |
| Git | Git 提供代码版本和仓库状态信息；本系统只记录与本次测试相关的信息，不改变代码版本或仓库内容。 |
| 算法源码仓库 | 算法仓库提供源码以及算法自身的构建、运行和输出要求；本系统使用这些信息完成测试，但不修改算法实现。 |
| 本地数据集 | 数据集提供算法测试所需的输入及必要的参考信息；本系统使用并记录本次选择的数据集，但不改变原始数据内容。 |

## 6. 非功能需求

- **可扩展性：** 系统应能够在不影响已有算法和历史结果的前提下接入新的定位算法。
- **可追溯性：** 每次测试结果应能够追溯到对应的算法、代码版本、测试数据和评估要求。
- **可重复性：** 对相同的算法版本、测试数据和测试要求，系统应能够重复执行同一基线测试流程。
- **可靠性：** 默认模式下，单个测试数据发生问题时系统应记录未运行成功并继续后续数据；用户应能够启用首次失败立即退出的人工排错模式；两种模式都必须保留已经完成的有效结果并明确说明未完成内容。
- **数据保留：** 历史测试结果不应被新的测试覆盖，除非用户明确删除。
- **易用性：** 用户应能够通过清晰的终端操作完成配置、启动、查看结果和发起对比。
- **可理解性：** 报告中的指标、状态和结论应具有明确含义，使用户能够判断本次测试及算法变化的影响。

## 7. 验收标准

1. 系统必须能够接收一个指定定位算法的构建要求、运行要求、输出说明和评估要求。
2. 系统必须能够记录本次测试对应的代码版本和代码仓库状态。
3. 系统必须能够完成算法构建，并明确报告构建是否成功。
4. 系统必须能够选择一个或多个本地测试数据，并明确本次测试实际包含的数据范围。
5. 系统必须能够对所选测试数据运行算法，并分别记录每个测试数据的运行结果。
6. 系统必须能够对运行成功且输出有效的算法结果完成性能评估，并保存对应指标。
7. 系统必须能够保存一次测试的代码版本、数据集、运行结果和评估结果，且后续测试不会自动覆盖这些内容。
8. 系统必须能够将本次结果与基线结果或历史结果进行回归对比，并说明指标变化。
9. 系统必须能够生成包含测试范围、执行结果、性能指标和对比结论的报告。
10. 系统必须能够识别并说明构建、运行或评估未完成的情况，同时保留已经产生的有效结果；默认模式继续后续数据，首次失败退出模式停止后续数据。
11. 系统必须能够在不修改已有算法接入内容和历史结果的前提下接入另一个符合要求的定位算法。
12. 用户发起一次基线测试后，系统必须能够完成从构建到报告的完整流程，不要求用户逐个测试数据重复操作。

## 8. 术语表

| 术语 | 含义 |
| --- | --- |
| 基线测试 | 在确定的算法版本、测试数据和评估要求下执行的一次完整测试，用于建立或验证算法性能。 |
| run | 用户发起的一次完整基线测试过程。 |
| dataset | 用于运行和评估定位算法的一组测试数据。 |
| baseline | 被选作长期参考的测试结果。 |
| 历史结果 | 系统过去保存的基线测试结果。 |
| 回归对比 | 将当前结果与参考结果比较，判断算法性能指标发生了哪些变化。 |
| voeval | 负责根据算法输出计算定位性能指标的评估工具。 |
| 算法接入 | 为使一个定位算法能够被系统构建、运行、评估和展示结果而提供必要信息的过程。 |
