# 算法基线测试系统——调研简报（2026-07-14）

## 1. 调研目标

1. 确认成熟 benchmark 及 SLAM、VO 方案如何比较不同算法，以及如何比较同一算法的不同代码版本。
2. 明确结果可比的前提、指标变化的表达方式，并提炼对算法契约和回归评测的启发。

## 2. 调研范围

### 2.1 通用 benchmark 与性能回归框架

#### [ASV](https://asv.readthedocs.io/en/stable/using.html#comparing-the-benchmarking-results-for-two-revisions)

- 同一 benchmark 可在两个 Git 版本间比较基线值、候选值和比值，benchmark 定义变化后旧结果不再参与比较；不同算法只有被表达为同一 benchmark 的参数且运行条件一致时才适合横向查看。它不负责 SLAM 工程的构建、数据运行和 VO 评估。
- 报告展示指标：执行时间、对象内存、进程峰值内存或用户定义数值；版本报告补充基线值、候选值、绝对变化、变化率、倍数、相对不确定度和对应提交。

#### [Conbench](https://conbench.github.io/conbench/pages/benchmarking_and_the_dev_process.html)

- 同一 benchmark 通过 Git 关系选择基线，并根据相同历史结果的 z-score 判断改善或回归；不同实现可以作为不同 case 保存，但只有仓库、硬件、case 和上下文一致时才比较。其结果契约值得借鉴，但完整服务体系不适合当前本地 CLI 项目。
- 报告展示指标：耗时、内存或数据量、吞吐量；单项结果包含样本数、均值、标准误、中位数、最小值、最大值、标准差、四分位数和四分位距，版本对比包含基线值、候选值、百分比变化和 z-score。

### 2.2 SLAM、VO 运行与评测框架

TUM、KITTI、EuRoC 主要提供数据集、格式或评测规范，不是完整测试系统，因此不作为独立方案比较。

#### [SLAM Hive](https://github.com/SLAM-Hive/slam_hive_web)

- 平台将算法运行、单任务评估和多任务分析分开，并可比较同一数据集上的多个算法；同一算法的代码版本只能注册成不同任务来模拟，系统没有 Git 提交与基线关系，也不会自动计算版本变化率或回归结论。
- 报告展示指标：ATE 和 RPE 的 RMSE、均值、中位数、标准差、最小值、最大值及误差平方和，以及 CPU 最大值、CPU 均值和内存最大值；同时展示轨迹、误差和资源使用图。

#### [SLAMBench](https://github.com/pamela-project/slambench)

- 通过统一生命周期加载多个算法库，在同一输入和真值下比较 ATE、RPE、耗时、内存和功耗；同一算法的两个构建版本也可作为不同库加载，但 Git 历史和基线需要外部记录，框架不提供版本变化率和自动回归判定。
- 报告展示指标：ATE、RPE、每帧总耗时及预处理、跟踪、融合、光线投射、渲染等阶段耗时，CPU 和 GPU 内存、功耗或能耗；同时列出实验参数和逐帧结果。

### 2.3 构建与回归测试中的插件化思路

- [pytest 插件](https://docs.pytest.org/en/stable/how-to/writing_plugins.html)由核心声明扩展点，插件只实现需要参与的能力；[Bazel 规则](https://bazel.build/extending/rules)要求任务明确声明输入、输出和执行动作，二者都把扩展差异限制在公共流程之外。
- 对本项目的启发是：公共 Pipeline 先定义稳定能力和调用时机，算法接入只实现自身差异；构建输入、构建产物、运行结果和成功条件都应被明确声明和验证。

## 3. 横向对比

| 项目 | 多算法接入 | 接入方式 | 历史管理 | 回归对比 | 报告 |
| --- | --- | --- | --- | --- | --- |
| ASV | 非原生，可作为同一 benchmark 的不同参数 | benchmark 定义或插件 | 关联 Git 版本、机器和环境 | 同一测量项跨版本比较基线值、候选值和比值，并按阈值分组 | 耗时、对象内存、峰值内存或自定义数值，以及当前值、变化率、倍数和趋势 |
| Conbench | 支持不同实现和 case | 适配器转换为统一结果 | 保存 run、Git、硬件、上下文和历史分布 | 相同可比项使用基线与 z-score 判断变化，不同 case 分开比较 | 耗时、内存、吞吐量，以及统计量、百分比变化和 z-score |
| SLAM Hive | 支持多个算法和参数配置 | 运行环境、参数模板和适配脚本 | 保存任务、配置和结果，无 Git 版本关系 | 同数据集原始指标并排；代码版本需注册成不同任务，不能自动计算变化率 | ATE、RPE 统计量、CPU、内存、轨迹图和误差图 |
| SLAMBench | 支持多个算法库 | 统一生命周期的共享库 | 保存本次结果，无 Git 与基线历史 | 同输入和真值下并排指标；代码版本需加载两个构建，不能自动判定回归 | ATE、RPE、分阶段耗时、CPU 和 GPU 内存、功耗或能耗 |

## 4. 对本项目的启发

### 4.1 算法契约设计参考

- SLAMBench 的统一生命周期证明公共流程可以只依赖稳定阶段，但共享库接口也会迫使算法采用同一进程模型；因此本项目应统一能力和调用时机，而不是统一语言、进程或算法内部结构。
- SLAM Hive 用脚本和约定目录获得了较强兼容性，但也把真实契约藏进文件名、路径和完成标志；结合 Bazel 的显式输入输出思想，插件必须声明所需输入、预期输出和成功条件，具体命令与目录仍留在插件内部。
- SLAM Hive 将算法、数据集、任务和结果分开，说明单个数据集应成为运行、评估和保存的最小边界；Pipeline 负责选择和遍历数据集，插件不能接管整次 run。
- ASV 会在评测定义变化后停止比较旧结果，说明算法 commit、插件契约版本、指标定义版本和 run 身份必须分开；任何一项变化都不能在指标同名时被静默忽略。
- 每次 run 应保留当时实际使用的代码状态、构建产物来源、插件、参数、数据集、运行环境和评估规则；插件升级后，历史结果仍按原快照解释，不能被新规则改写。

### 4.2 要避免的坑

- SLAMBench、SLAM Hive 的多算法同场比较与 ASV、Conbench 的跨版本回归是两种实验；前者回答算法间差异，后者回答代码修改影响，不能共用一套“改进或退化”结论。
- Conbench 只比较仓库、硬件、case 和上下文一致的结果，SLAM 平台则要求相同数据集和真值；本项目也必须先检查算法身份、数据集、参数、指标契约、评估版本和必要运行条件，再允许比较。
- “上一版”不能简单理解为时间上最近的结果；它应是通过可比性检查后最近一次成功的 run。基线选择属于 Pipeline，并需保留选择的是上一成功结果还是用户指定结果。
- 算法原始输出、可评估输出、单数据集指标和整次 run 汇总是四种结果；每层分别判断有效性，不能仅凭进程退出或文件存在就认定算法成功。
- 指标契约除名称和值外，还要包含含义、单位、好坏方向、有效条件、对齐或尺度条件和覆盖范围；轨迹提前结束时 ATE 可能反而变小，因此精度变化必须和输出完整性一起看。
- 跨数据集只汇总可比指标相对基线的变化比例，不计算不同量级原始值的平均值；失效值不参与百分比统计，但必须保留数量和对应数据集。
- 算法运行失败、指标失效和性能退化是不同结论；成功变失败应体现可用性退化，失败值不能记为零，构建、评估或系统问题也不能计入算法失败阈值。
- ASV 和 Conbench 都把测量波动纳入回归判断；如果没有重复样本，本项目只能报告观察到的百分比变化和阈值结果，不能把一次变化描述成统计显著。资源测量还应与 voeval 和报告生成分开，避免测量过程污染耗时和内存。

### 4.3 可以复用的部分

- 借鉴 SLAM Hive 的运行、评估、分析分层以及 ASV、Conbench 的原始测量与派生比较分离：算法原始输出、voeval 单数据集结果和 Pipeline 对比报告分别保留，从而可以不重跑算法就重新评估、换基线或生成报告。
- 复用的是 SLAMBench 的生命周期、SLAM Hive 的对象分离、ASV 的评测定义版本和 Conbench 的可比上下文；不照搬共享库接口、固定容器任务或服务端体系，也不把另一套评估工具引入 Pipeline。

### 4.4 本项目的差异化

- 通用回归框架不负责 SLAM 运行，SLAM 平台又缺少 Git 版本回归；本项目的核心对象因此不是 commit 或单项指标，而是把代码状态、构建产物、插件快照、数据集输出、voeval 指标和基线关系绑定起来的一次 run。
- Pipeline 报告只汇总整次 run 的成功与失败、逐数据集结果和版本变化；轨迹、误差分布等单数据集细节继续由 voeval 展示，二者共享结果但不合并职责。
